# ADR-0038 — A daemon control socket exposes live introspection (the `status.query` RPC)

- **Status**: Proposed (accepted at the next graduation, per the ADR-0031/0036 precedent)
- **Date**: 2026-06-20
- **Slice**: Spec B G6-2b-2c — `docs/superpowers/specs/2026-06-13-comms-gateway-resume-design.md`
- **Relates to**: [ADR-0031](0031-comms-socket-transport-for-the-foreground-tui.md) (the sibling 0600 local-socket carrier whose peer-auth + bind discipline this reuses), [ADR-0036](0036-gateway-adapter-hosting-inversion.md) (Spec B G6-1 — names "the alfred status snapshot" as a downstream consumer), issue #288 (the gateway program)
- **Supersedes**: the withdrawn file-snapshot approach (PR #299 — never merged)

## Context

The daemon holds the live per-adapter status the operator wants from `alfred daemon status` (and, later, `alfred status` / `alfred gateway adapters --wait-ready`): the `AdapterStatusObserver.all_latest()` map (the latest accepted `gateway.adapter.*` transition per adapter) and the `CrashIncidentReconciler` (correlated crash incidents per incarnation). Both are in-process, held for the daemon's lifetime in the `_CommsBootGraph`.

The CLI cannot reach them. `alfred daemon status` reads only the pidfile; `alfred status` reads only Settings + the secret broker. Neither dials the daemon. The only socket the daemon runs is the per-adapter comms wire ([ADR-0031](0031-comms-socket-transport-for-the-foreground-tui.md)) — a **one-shot accept, bidirectional notification pump** bound to a single TUI peer. That is the wrong shape for a query: it is a long-lived conversation pump, not a request/response service, and it is consumed by the foreground TUI.

A prior attempt (PR #299) published a status **file** the CLI read. It was rejected as hacky: a file has a staleness window (a periodic publisher races the reader), needs a reap-of-stale-state path, and needs a `boot_id` cross-check that is anti-stale but not anti-forgery. The principled replacement is a **live query**: one source of truth (the in-process observer/reconciler), read at query time, so there is no staleness, no publisher, no reap, and no `boot_id` caveat — the daemon answering the socket IS, by construction, the live daemon.

## Decision

**Decision 1 — A dedicated request/response control socket, separate from the comms wire.** The daemon binds `~/.run/alfred/control.sock` (`0600`, under the already-`0700` runtime dir) and serves a small **request/response JSON line protocol**: a CLI dials, sends one `ControlRequest` (`{version, id, method, params}`), the daemon authenticates the peer, routes the method to a handler that builds the response **live**, writes one `ControlResponse` (`{id, result|error}`), and closes. Unlike the comms socket, the control socket is **multi-connection, request/response, stateless-per-connection** — a true control plane.

**Decision 2 — Reuse the ADR-0031 socket security via a shared module; do not fork it.** The peer-uid auth (`SO_PEERCRED`), the owner-only bind (mkdir-0700 + unconditional dir-chmod + unlink-stale + bind + chmod-0600 + listen), the call-time runtime-dir resolution, and the pre-dial `lstat` owner-backstop are EXTRACTED into `src/alfred/plugins/_local_socket.py` and reused by BOTH the comms socket and the control socket. One audited implementation, not two divergent copies (and the "three independent `~/.run/alfred` derivations" drift is retired). The breadcrumb log prefix is parameterised so the comms wire keeps `comms.socket.*` and the control plane uses `daemon.control.*`.

**Decision 3 — The protocol module is the introspection contract v1, transport-agnostic.** `_daemon_control_protocol.py` is the long-lived asset: the method schema + frozen Pydantic request/response models a FUTURE remote management plane (dashboards / ops) will front over an authenticated HTTP transport (see `docs/ARCHITECTURE.md` D1 — the management-plane transport is an explicitly DEFERRED decision). The LOCAL CLI fronts the SAME contract over the unix socket TODAY. There is NO transport coupling in the models — they are pure data. **This slice builds only the local unix-socket front; no remote/HTTP surface is in scope.**

**Decision 4 — Auth is degrade-open, justified ONLY by read-only + non-sensitive + FS-perms-of-record.** On a `SO_PEERCRED` host a different-uid dial is refused (and audited — see Decision 6); on a host without `SO_PEERCRED` (a macOS dev box) the peer-uid check degrades OPEN to the FS-perms-of-record (the `0600` socket under the `0700` dir whose parent is the operator's owner-only home is the same-uid enforcement). This is acceptable here ONLY because the control plane is **read-only and non-sensitive**: the response carries operational metadata (adapter_id, state, occurred_at, incarnation, incident count + the latest incident's seq/source/id) and never secret/T3 content. **Any future MUTATING method, or any method returning a sensitive field, RE-OPENS this auth decision** — it would likely be fail-closed (refuse on a no-`SO_PEERCRED` host) plus a step-up, and MUST be reasoned through on its own terms, not inherited from this read-only grant.

**Decision 5 — DoS-bounded serving.** The control plane is multi-connection, so each per-connection exchange (authenticate → read → respond) runs under an `asyncio.timeout` (~5 s) — a peer that connects and never writes a newline cannot hold a serve task + fd forever. Live serve tasks are gated by a bounded `asyncio.Semaphore` (24); past the ceiling a connection is closed immediately, not queued (the kernel `backlog` bounds only the accept queue, not the number of live tasks). A malformed / over-bound / unknown-method request on one connection is answered (or loud-closed) WITHOUT wedging the accept loop — a control-plane outage would blind the operator worse than a single bad request.

**Decision 6 — Read-not-audited, reject-audited; loud on a failed reject-audit.** A read-only `status.query` writes NO audit row (a pure read — convention). A **peer-auth rejection** (a refused different-uid dial) IS audited: a loud `daemon.control.peer_uid_rejected` row via the dedicated `DAEMON_CONTROL_PEER_REJECTED_FIELDS` schema (no `adapter_id` — the control plane is daemon-global, not adapter-keyed). A rejection does not refuse the boot (an attacker racing the socket must not be able to DoS the daemon); the server keeps serving. But a FAILED audit-write of a reject is a security event (CLAUDE.md hard rule #7): it ESCALATES loud, distinct from the resilient-connection swallow — it is NOT folded into the generic per-connection fault handler.

**Decision 7 — Fail-closed framing; never `str(exc)` on the wire.** An over-bound request line raises BEFORE writing (the peer sees EOF — a silent close — not a partial frame). A malformed request answers an `error` token carrying the request id sentinel `UNKNOWN_REQUEST_ID = "?"` (the id is unknowable on a parse failure). A handler fault answers an `error` token carrying ONLY `type(exc).__name__` — never `str(exc)`, so no exception message ever reaches the wire. Every wire model is `extra="forbid"` with `max_length`-bounded peer-controlled fields (`method`, `id`), locked by a structural field-set test.

## Consequences

- **One source of truth, no staleness.** The response is computed at query time from the live observer/reconciler — no snapshot, no periodic publisher, no reap-of-stale-file, no `boot_id` "anti-stale-not-anti-forgery" caveat. The `0600` socket + `SO_PEERCRED` are the authenticity guarantee the file's non-secret `boot_id` could never be.
- **A new same-uid local IPC surface.** Its auth is `SO_PEERCRED` + the `0600` owner-only bind; a refused different-uid dial is audited. There is no network ingress and no cross-uid reach. The bind→chmod TOCTOU window is unchanged from ADR-0031 and is closed by the `0700` parent: the only writer to the runtime dir is the daemon's own uid, so an attacker cannot plant an inode between the bind and the chmod.
- **The channel generalises (G6-5 substrate).** The method router takes `gateway.adapters` next, and `alfred gateway adapters --wait-ready <adapter>` (a live-ness consumer the file served poorly) reuses this exact channel. The RECOMMENDED `--wait-ready` shape is a **client-side poll loop over repeated `status.query`** — it keeps the server stateless and puts the 0/1/2/3 exit-code contract in the client. The alternative (a long-lived-connection handler that awaits an observer transition) is rejected for this slice: it would make the server stateful and re-open the exchange-timeout/concurrency reasoning per long-poll. Handlers are nonetheless typed `async` now so the blocking-readiness handler does not re-type the seam.
- **`params` is reserved, not yet consumed.** `status.query` ignores `params`. Any future `params`-consuming method MUST validate `params` into a per-method frozen Pydantic model, never read the raw dict (the same `extra="forbid"` discipline the request envelope already enforces).
- **The file approach is retired.** PR #299's `_daemon_status_snapshot.py` / `_daemon_status_publisher.py` are never built; the live query is the single status-reachability path.

## Alternatives considered

- **The status file (PR #299).** Rejected: staleness window, reap-of-stale-state, `boot_id` is anti-stale not anti-forgery, and it is a throwaway the moment `--wait-ready` needs live-ness. The maintainer rejected it as hacky; this ADR is the principled replacement.
- **Reusing the comms socket (ADR-0031).** Rejected: wrong shape (a one-shot bidirectional notification pump, not a query service) and TUI-bound (consumed by the foreground `alfred chat`). A query plane needs multi-connection request/response.
- **A remote/HTTP management surface now.** Out of scope: the management-plane transport is a DEFERRED decision (`docs/ARCHITECTURE.md` D1). This slice ships only the local unix-socket front of the transport-agnostic contract.
