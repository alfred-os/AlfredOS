# Slice 2 — PR C: SecretBroker file backend + perf-006 redactor caching — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development`. Steps use checkbox (`- [ ]`) syntax — tick them as you go so a follow-up session can resume without re-deriving state.

**Goal:** Land the file-backed `SecretBroker` at `~/.config/alfred/secrets.toml` with fail-closed POSIX permissions, four typed config-error subtypes, a `.git`-in-parent-walk refusal, and a bounded redactor regex cache (perf-006). After this PR merges, the broker can read `discord_bot_token` from disk, any direct `os.environ` access to a `SUPPORTED_SECRETS` key fails CI via the new AST-scan test, and `SecretBroker.redact` is O(1)-amortised across calls instead of rebuilding its alternation on every invocation.

**Spec:** [`docs/superpowers/specs/2026-05-26-slice-2-discord-multiuser-design.md`](../specs/2026-05-26-slice-2-discord-multiuser-design.md) — §2 SecretBroker file backend (lines 183-215), §3 secret-broker i18n keys (line 532), §5 unit + integration tests (lines 793, 795, 804), §6 PR C row (line 871), §7 perf-006 punch list (line 903).

**Depends on:** PR A merged to `main` (identity layer + ADR-0012 placeholder + ContextVar i18n). PR B may not be merged yet — this PR is independent of any budget / memory / orchestrator changes.

**Architecture:** Extend `src/alfred/security/secrets.py` with two new constructor kwargs (`secrets_file: Path | None`, `require_file: bool`) and a layered path-resolution pipeline (host default → container env override → caller override). Add a `SecretBrokerConfigError` base with three concrete subtypes (`SecretBrokerPermissionsError`, `SecretBrokerFileMissingError`, `SecretBrokerNotAFileError`) so CLI top-level dispatch can `except SecretBrokerConfigError` once while error-message i18n routing branches on the subtype. Declare `_PREFER_FILE: frozenset[str] = frozenset({"discord_bot_token"})` next to `SUPPORTED_SECRETS` so `get()` precedence is explicit at the data-source level — env wins for slice-1 keys (backward-compat), file wins for slice-2+ keys. The permissions check is fail-closed at init time: not a symlink, owner matches `os.getuid()`, mode `& 0o077 == 0`, parent directory not group/world-writable. Broker also walks parent dirs and refuses if it finds a `.git` directory (configurable off via `allow_inside_git_worktree=True` for tests). New AST-scan test `tests/unit/security/test_no_direct_env_reads.py` fails the build if any module other than `src/alfred/security/secrets.py` reads `os.environ["ALFRED_*"]` for a `SUPPORTED_SECRETS` key; failure messages share a `_remediation_message(...)` helper at `tests/unit/_shared/import_violation.py` with PR D1's `test_no_direct_adapter_imports.py` (te-004). Redactor caching (perf-006): compile the alternation regex once, invalidate on secret mutation via an internal version counter, cap at `MAX_REDACTOR_PATTERNS = 256` distinct values with a one-shot WARN + `alfred_redactor_pattern_overflow_total` counter bump on overflow (degraded-but-non-failing fallback to a smaller regex).

**Tech Stack:** Python 3.12+ • `tomllib` (stdlib) • `pathlib` • `stat` • `re` • `prometheus_client` (slice-1 dep) • `structlog` (slice-1 dep) • pytest + testcontainers + `hypothesis` (where regex-cache invariants warrant property tests) • coverage with per-file `--cov-fail-under=100 --cov=src/alfred/security/secrets.py`.

**Subagent owner:** `alfred-security-engineer` — every task in this PR sits on the trust boundary (secret-store ingestion, redactor surface, CLI-bootstrap fail-closed semantics). Conventions pass-through: `alfred-python-developer` reviews the final diff for type strictness, immutable defaults, and `t()` discipline before submission.

| Layer | Setting | Resolves to | Used by |
| --- | --- | --- | --- |
| Host default | `Settings.secrets_file` (Pydantic) | `Path.home() / ".config/alfred/secrets.toml"` | `alfred` CLI run on the host |

Test names follow the existing slice-1 pattern (`test_<behaviour>__<context>`).

**`tests/unit/security/test_no_direct_env_reads.py` (new).** AST-scans every `src/alfred/**/*.py` file. Walks each module's AST looking for `ast.Subscript` nodes whose `value` is `ast.Attribute` (`os.environ`) or `ast.Name` (`environ` imported from os), and whose `slice` is a string literal matching `r"^ALFRED_[A-Z0-9_]+$"`. For each match, it derives the secret name (lowercase, drop `ALFRED_` prefix) and asserts the name is not in `SUPPORTED_SECRETS` — unless the file IS `src/alfred/security/secrets.py` (the one legitimate reader). Also matches `os.environ.get("ALFRED_…")` (`ast.Call` of `ast.Attribute` `os.environ.get`). Violations are collected, the test fails through `_remediation_message(violations, category="env_read")`. Property: the test imports `SUPPORTED_SECRETS` from `alfred.security.secrets` at test time so the matcher tracks the broker's own truth — adding a new secret to the set automatically expands the no-direct-reads guard with zero test edits. ADR-0012 references this test by file path so the invariant is discoverable from the design doc.

**`tests/unit/security/test_redact_perf.py` (new).** Targets the perf-006 invariants in isolation so the broader `test_secrets.py` doesn't grow a perf-shaped tail. Cases: (a) two calls to `redact()` with the same secret set call `re.compile` exactly once; (b) mutating the secret set bumps an internal version counter and the next `redact()` recompiles; (c) `MAX_REDACTOR_PATTERNS=256` boundary — at 256 patterns the regex still compiles; at 257 the broker logs one WARN + bumps `alfred_redactor_pattern_overflow_total` + falls back to the most-recently-added-256 patterns (or the 256-longest — decision pinned in §3 below); (d) the WARN is one-shot per overflow event, not per call (assert via `caplog` count after 10 consecutive overflowing `redact()` calls); (e) hypothesis property test — for any input text and any secret set ≤256, `redact(redact(text)) == redact(text)` (idempotence under cache pressure).

**`tests/integration/test_secrets_file_backend.py` (new).** Spec §5 line 804. Boots a tmp-path fixture (no testcontainer needed — broker is pure filesystem + env): writes `secrets.toml` with `chmod 600`, ensures parent is `chmod 700`, asserts the broker loads `discord_bot_token` from the file even when `ALFRED_DISCORD_BOT_TOKEN` is also set in the env (file-prefer for `_PREFER_FILE`); asserts the broker loads `deepseek_api_key` from the env even when the file also defines it (env-prefer for slice-1 keys); asserts the broker rejects a path with a `.git` directory in the parent walk; asserts `require_file=True` raises `SecretBrokerFileMissingError` when the path doesn't exist. The integration variant complements the unit suite by exercising the real filesystem (`Path.stat`, real `os.chmod`, real `tomllib.load`) end-to-end so any platform-specific stat quirk surfaces in CI rather than at deploy time.

**No source file outside `src/alfred/security/secrets.py` is modified by this PR.** The `Settings.secrets_file` Pydantic field already lands in PR A's settings refactor (see ADR-0012 placeholder for the contract); PR C consumes it. - [ ] **Task 4 — `_validate_secrets_file_security` fail-closed permissions check.**
  Subagent: `alfred-security-engineer`.
  Failing tests first (one per check, four total): `::test_perms_check__rejects_symlink`, `::test_perms_check__rejects_wrong_owner`, `::test_perms_check__rejects_world_readable_file`, `::test_perms_check__rejects_group_writable_parent`. Each test sets up the offending condition under tmp_path (chmod, symlink, etc.) and asserts `SecretBrokerPermissionsError` is raised with the offending `mode` populated in octal.
  Then implement: `_validate_secrets_file_security(path: Path) -> None` private function. Stats the path with `os.lstat` (not `stat` — `lstat` doesn't follow the symlink, so the `S_ISLNK` check works). Order of checks: (1) `stat.S_ISLNK(mode)` → reject symlink; (2) `st_uid == os.getuid()` → reject wrong owner; (3) `st_mode & 0o077 == 0` → reject group/world-readable file; (4) `path.parent.stat().st_mode & 0o022 == 0` → reject group/world-writable parent. Each failure raises `SecretBrokerPermissionsError(path=path, mode=mode, parent=path.parent)` with the mode populated as the integer (the i18n template renders it as `oct(mode)`). The check is called from `__init__` IF the resolved file path exists; the missing-file case is handled in Task 6.
  Conventions: function is pure-of-side-effects-except-raise; no logging inside (the caller catches at the boundary and logs). POSIX-ACL non-coverage is documented in the function docstring referencing ADR-0012's "known gap" section.

- [ ] **Task 5 — `.git`-in-parent-walk rejection.**
  Subagent: `alfred-security-engineer`.
  Failing tests first: `::test_git_walk__rejects_path_inside_worktree` and `::test_git_walk__allow_inside_git_worktree_bypasses`. Both create a tmp-path layout with a `.git` directory at the parent or grandparent of the secrets file.
  Then implement: `_walk_for_git_parent(path: Path, max_depth: int = 12) -> Path | None` private function. Walks from `path.parent` upward up to `max_depth` levels (defends against pathological symlink loops; 12 is well past any sane host depth — `/`, `/home`, `/home/user`, `~/.config`, `~/.config/alfred` is already 5). Returns the first ancestor that contains a `.git` directory, or `None` if none found. The constructor calls this iff `allow_inside_git_worktree=False`; if it returns non-None, raises `SecretBrokerPermissionsError(path=path, parent=found_git_dir.parent, mode=0)` — mode is 0 because this isn't a perm-bits failure, it's a location failure; the i18n template branches on `parent is not None and mode == 0` to render the .git-specific message (folded into `secrets.file_perms_too_open`; we deliberately do NOT add a fourth i18n key — three keys is the spec count).
  Conventions: `max_depth` is an explicit parameter, not a magic number. The walk uses `path.parent` iteration; stops at the filesystem root (`path.parent == path`). No recursion (iterative for clarity).

## 3. Open questions / decisions deferred to plan time

Two micro-decisions the planner pins now so the agent doesn't burn a turn re-deriving them:

**Decision 3.1 — `MAX_REDACTOR_PATTERNS=256` overflow selection rule.** When the broker holds more than 256 distinct secret values, the redactor compiles only 256 of them into the alternation. Two candidate selection rules:

- (a) **256 longest values.** Aligns with the slice-1 "longer-first" invariant — the longest values are the highest-information-content leakage risk, so they're the right ones to keep. Dropping short tail values means short cleartext fragments may leak, but a 6-character token is far less valuable to an attacker than a 60-character one. **This is the rule the plan ships.**
- (b) Most-recently-added 256 values. Simpler implementation but less defensible — it would silently drop a long-standing high-value secret in favour of a freshly-added short one.

The plan picks (a). The decision is recorded inline in `secrets.py`'s `redact()` docstring with a one-line rationale; ADR-0012's full body (PR E) re-states it. The `test_redact_perf.py` overflow case asserts the broker keeps the longest 256, not the most-recent.

**Decision 3.2 — `.git`-walk reuses `SecretBrokerPermissionsError` vs. its own subtype.** Spec §2 line 215 says "the broker walks parent dirs at init; refuses if it finds a `.git` directory". The error subtype isn't pinned. Two options:

- (a) **Reuse `SecretBrokerPermissionsError` with `mode=0` as a sentinel** for "this is a location failure, not a perm-bits failure". The i18n template branches on `mode == 0 and parent is not None and (parent / ".git").is_dir()` to render the `.git`-specific message. Keeps the spec's three-key count; one fewer exception class to document.
- (b) Add a fourth subtype `SecretBrokerInsideGitWorktreeError(SecretBrokerConfigError)` — cleaner taxonomy but exceeds the spec's three-key i18n target and the spec's "three concrete subtypes" wording at line 211.

The plan picks (a) to honour the spec's explicit "three concrete subtypes" + three-i18n-keys count. If the security review during PR C wants the cleaner taxonomy, that's a one-line spec amendment + an extra i18n key in PR D2 or PR E; PR C ships (a).

No other open questions — the spec is complete enough that every implementation detail above is sourced from it.
