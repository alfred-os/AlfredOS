# Slice 2.5 — PR-B: `memory.episodic.record` hooks proof-of-concept — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development`. Steps use checkbox (`- [ ]`) syntax — tick them as you go so a follow-up session can resume without re-deriving state. This is trust-boundary work (the hooks subsystem is the in-band capability gate, and `record` is a persistence action) — TDD is HARD here, not advisory.

**Goal:** Instrument the single proof-of-concept action `EpisodicMemory.record` with the five internal hookpoints the spec specifies (two `pre`, one `error`, one `post`, one `cancel`), threaded through PR-A's `invoking()` helper. Introduce the `EpisodicRecordInput` frozen dataclass mirroring `record`'s ten kwargs (including `language` per i18n rule #3, and `trust_tier`). Wire the dispatch engine's `AuditSink` to the real `audit.append` path so every hook-trace fault row persists. Add the adversarial `hooks` corpus category and its schema additions. After this PR merges: with zero subscribers the persisted `Episode` row is byte-identical to today's (golden-row regression test guards this); a system-tier redactor registered on `before_db_write` scrubs a leaked secret from `content` before the write (integration test); and a user-plugin attempting system-tier registration on a system-only hookpoint is refused and audited (adversarial payload).

**Architecture:** Two locus-of-change files. The post hookpoint is **`after_flush`, not `committed`** (mem-1, see Decision 3.1) — `record` only flushes; the durable COMMIT is the caller's `session_scope` transaction (`db.py` lines 80-94, wrapping the orchestrator turn at `core.py` line 255). **`src/alfred/memory/hooks_audit_sink.py` (new).** A thin adapter `EpisodicAuditSink` implementing PR-A's `AuditSink` Protocol by delegating to `AuditWriter.append`. Maps a hook-trace event (`hooks.subscriber_error` / `hooks.chain_timeout` / `hooks.unauthorized_refusal` / `hooks.reentry_bypass` / `hooks.error_suppressed` / refusal rows) to the `audit.append` keyword args: `event=<hook event key>`, `actor_user_id=<from ctx.input.user_id when present, else None>`, `subject={correlation_id, hookpoint, kind, tier, detail}`, `trust_tier_of_trigger=<ctx.input.trust_tier>`, `result=<the fault disposition: "refused"/"fault"/"bypass">`, `trace_id=ctx.correlation_id`, `persona_id=ctx.input.persona_id`, `language=ctx.input.language`, **`cost_estimate_usd=0.0`** (the real `AuditWriter.append` at `src/alfred/audit/log.py:46-60` declares `cost_estimate_usd: float` keyword-only with **no default** — a hook-trace row carries no provider cost, so the adapter passes `0.0` explicitly; omitting it makes the `append(...)` call invalid — TE-1). **Signature-parity guard (recommended — mirrors Task 1's `EpisodicRecordInput` drift-guard):** a reflective check that the adapter supplies every keyword-only required parameter of `inspect.signature(AuditWriter.append)`, so a future kwarg added to `append` without a matching adapter update fails CI rather than drifting silently. **No recursion** (spec §6.8): this sink calls `audit.append`, and `audit.append` does NOT call `memory.episodic.record`, so a hook-trace row does not re-enter the instrumented action. The adapter is the seam where the PoC opts the dispatch engine out of the structlog-only default sink (PR-A's `AuditSink` Protocol default) and into the durable audit path. Lives in `memory/` (not `hooks/`) because it depends on `alfred.audit.log` and the PoC owns the wiring — keeping the hooks subsystem free of an audit-writer import preserves PR-A's dependency direction. **Fault rows are written on a FRESH, short-lived session — NOT the turn's session (memB-1, Decision 3.6).** A `write_failed` (error) chain fires precisely when `_persist`'s `flush()` raised, which marks the turn's `AsyncSession` failed-state; because `AuditWriter.append` shares the same `session_scope` factory (`core.py:225/255`), appending the fault row on the poisoned session would raise `InvalidRequestError` — the fault row would never persist, defeating the durability claim (Decision 3.2) and tripping CLAUDE.md rule #7. So `EpisodicAuditSink` opens its **own short-lived session/connection** (its own `session_scope`, independent of the turn's) for each fault-row append, so hook-trace audit persists even when the action's transaction is rolling back.

**`tests/integration/memory/test_episodic_hooks_poc.py` (new).** The PoC integration test (spec §7 final paragraph). Real Postgres via testcontainers (`alfred-memory-engineer` quality bar: integration tests use real Postgres). Cases:

The test comment must say "ORM Python-side defaults" (memB-2) so a future reader does not add a `server_default` migration expecting parity. - [ ] **Task 4 — Thread the two `pre` hookpoints (`before_validate`, `before_db_write`) via `invoking`.**
  Subagent: `alfred-memory-engineer`.
  Failing tests first (`test_episodic_hooks_wiring.py`): spy subscribers (registered via `fresh_registry()` + the granted `DevGate(allow_system=True)` where `system` tier is needed) assert `before_validate` fires with the constructed input, then `_validate` runs, then `before_db_write` fires with the (possibly mutated) input, then `_persist` runs with the input the `before_db_write` chain produced. A `before_db_write` subscriber that mutates `content` is observed by `_persist`; a `before_db_write` `HookRefusal` (system tier — the only `refusable_tier`) short-circuits so `_persist` never runs and no row is written.
  Then implement: rewrite `record` to:
  ```python
  inp = EpisodicRecordInput(user_id=user_id, role=role, content=content, trust_tier=trust_tier, ...)
  async with invoking("memory.episodic.record", inp) as flow:
      flow = await flow.pre("before_validate")            # all tiers, fail_closed=False (default)
      self._validate(flow.input)
      flow = await flow.pre(
          "before_db_write",
          subscribable_tiers={"system", "operator"},
          refusable_tiers={"system"},
          fail_closed=True,                                # security stage — a timed-out/erroring redactor MUST NOT write
      )
      async with flow.body():
          await self._persist(flow.input)
  ```
  (`_validate` is a small private guard — see Task 6; for now a no-op placeholder that the test will grow.) **The `flow.pre()` return contract is PINNED by PR-A Task 13** (no hedge): `await flow.pre(stage, **kw)` returns the **same flow object** and threads the `pre` chain's output frozen `HookContext` into the flow's internal `_ctx` holder, so reading `flow.input` immediately after `await flow.pre(...)` reflects a `before_db_write` subscriber's `with_input` mutation (the ctx stays frozen; the flow is the mutable holder). The desugaring above is therefore exact — `flow = await flow.pre(...)` reassigns to the same object and `_persist(flow.input)` sees the chain output. Do NOT re-derive or "adapt to whatever shipped": PR-A locks this; PR-B consumes it verbatim.
  Conventions: `before_db_write` MUST set `subscribable_tiers={"system","operator"}`, `refusable_tiers={"system"}`, `fail_closed=True` — these are the security-stage values from spec §7; getting them wrong is a trust-boundary defect. async-first; structured.

## 3. Open questions / decisions pinned at plan time

**Decision 3.1 — Post hookpoint is `after_flush`, not `committed` (mem-1).** `record` does `session.add()` + `session.flush()` and returns; the durable COMMIT happens later in the *caller's* transaction (`db.py` `session_scope` lines 80-94 wraps the whole orchestrator turn — `core.py` line 255). Firing a post hook named `committed` right after the flush would let a compliance/SIEM subscriber assert durability for a row a later same-turn failure can still roll back — a durability lie. `after_flush` is the truthful name. The rationale is baked into BOTH a code comment on `record` (Task 5) AND the golden-row test comment (Task 3) so a future reader cannot rename it to `committed` without confronting the reason. A real after-commit signal is a separate, later concern (it belongs on `session_scope`/the orchestrator turn, not on `record`).

**Decision 3.2 — AuditSink wired to the real `audit.append`; no recursion.** PR-B passes the real `AuditWriter.append` (via `EpisodicAuditSink`) as the dispatch engine's sink so `invoke()`'s fault rows (`subscriber_error` / `chain_timeout` / `unauthorized_refusal` / `reentry_bypass`) persist durably. **No recursion** because the PoC is `memory.episodic.record`, not `audit.append` — a hook-trace row is an ordinary `audit.append` that does NOT re-enter `record` (spec §6.8). Task 8 asserts an EXACT bounded audit-row count per `record`, proving no cascade. (Had the PoC been `audit.append`, the sink would need a chain-skipping `_append_raw`; choosing a non-audit PoC removes the hazard entirely — this is precisely why the spec rejected `audit.append` as the PoC.)

**Decision 3.3 — `EpisodicRecordInput.trust_tier` is publisher-asserted; the hooks layer does NOT re-verify T-tagging.** The orchestrator tags content T2 at the adapter boundary and passes `trust_tier=user_input_tier` to `record` (`core.py` lines 339, 346). `EpisodicRecordInput` carries that asserted tier verbatim. The hooks subsystem does not re-check tagging — data-classification tags per hookpoint (T1/T2/T3 matched against subscriber clearance) are explicitly Slice-3 (spec §6.10, blocking gate on out-of-process subscribers). This is stated so a reviewer does NOT flag the unverified `trust_tier` field as a tagging gap: it is a deliberate slice boundary, not an omission.

**Decision 3.4 — `ExpectedOutcome` needs NO enum edit.** The adversarial payload's `expected_outcome: refused` reuses the existing `refused` member of the `ExpectedOutcome` Literal (`payload_schema.py` line 52). Only the `Category` Literal, `_PREFIX_TO_CATEGORY`, `_ID_PATTERN`, and `conftest.py`'s `_CATEGORIES` tuple gain `hooks`/`hk`. This is the minimal schema surface for the new category — confirmed against the real `payload_schema.py`.

**Decision 3.5 — No new i18n catalog key from PR-B.** The PoC redactor mutates `content` silently (it does not surface a message). The only operator-facing hook strings (`HookError`/`HookRefusal` bodies, the `hooks.subscriber_must_be_async` decoration error) are PR-A's catalog keys. `locale/en/LC_MESSAGES/alfred.po` is listed in §0 only so a reviewer knows the i18n surface was considered and deliberately not extended. Task 9 carries a review checkpoint: if `_scrub` or the subscriber genuinely surfaces a new operator-facing string, the key + `pybabel` regen lands there (the gate at §2 covers it). The persisted-row `language` column IS exercised (Task 9's `language="ja-JP"` survives-mutation assertion) per i18n rule #3.

**Decision 3.6 — `EpisodicAuditSink` writes fault rows on a FRESH session, never the turn's poisoned session (memB-1).** A `write_failed` (error) chain fires exactly when `_persist`'s `flush()` raised — which marks the turn's `AsyncSession` failed-state. Since `AuditWriter.append` shares the same `session_scope` factory (`core.py:225/255`), appending the fault row on that poisoned session would raise `InvalidRequestError`: the durability claim (Decision 3.2) collapses and CLAUDE.md rule #7 (no silent failure in the audit path) is tripped — loudly, but with the fault row LOST. So the sink opens its own short-lived `session_scope` (independent connection) per fault-row append, so hook-trace audit persists even while the action's transaction rolls back. Stated in §0's `EpisodicAuditSink` design; proven by the real-flush-failure integration case (§0 integration-test list / Task 8) which forces `flush()` to raise and asserts the fault row still persists on the fresh session AND the no-unbounded-rows bound holds. (The original no-recursion case uses a `fail_closed=False` `before_validate` `ValueError` that never flushes — it never exercises the poisoned-session path, which is why this distinct case is required.)

No other open questions — the spec (§7, §6.3/§6.4, §8) and the confirmed `record` signature are complete enough that every implementation detail above is sourced.
