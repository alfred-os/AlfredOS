# G7-2.5 — The Live `web.fetch` Re-home (Spec C, #333)

**Status:** design — approved 2026-06-28; implementation pending (writing-plans next).
**Epic:** #333 (Spec C — egress control plane / connectivity-free core).
**Predecessor:** G7-2c (PRs #345/#346, merged `d782812d`) shipped the in-core relay client (C1), the §4.3 gate-checked extractor (C2), the tri-state dedup ledger, and the synthetic-driver proof.
**Successor / hard dependant:** G7-3 (the `internal:true` flip + deletion of the core's direct egress fallback) is **hard-blocked on G7-2.5 landing** — a live tool must not egress while the core's direct fallback still exists.

---

## 1. Goal

Re-home the live `web.fetch` tool egress out of the `alfred_web_fetch` plugin subprocess so the **gateway** performs the fetch via the shipped `EgressRelay`, and `dispatch_web_fetch` reaches the real relay. This is the **side-effecting cutover** that makes the G7-2c relay machinery a production path (its first production consumer). It is L-sized: it moves real enforcement (TLS/size/MIME/redirect) and the content-store/`ContentHandle` mint, not a "re-point."

**This slice does NOT** flip `internal:true`, delete the core's direct egress fallback, add kernel/DNS enforcement tests (all G7-3), harden Discord egress (G7-4), or rewrite PRD §5/§7.1 / ADR-0040 (G7-5, human-gated). It also does **not** add a live LLM caller — `dispatch_web_fetch` remains called only by tests until #339 wires the tool-calling loop after G7-3.

## 2. Locked context (decided in prior slices; not relitigated here)

- The connectivity-free core cannot open external sockets. Tool egress goes core → gateway `EgressRelay` over a length-prefixed JSON-frame wire. The gateway **fetches** the upstream and returns `EgressResponse(status: int, headers: Mapping[str, str], body: bytes)`.
- The **core** mints the T3 `ContentHandle` + tags T3 + runs the **one** sanctioned gate-checked seam `quarantined_to_structured(handle, schema, *, extractor, gate)` (gate-first; a `False` clearance raises `AlfredError`). The gateway holds no `CapabilityGateNonce` (ADR-0036), so it structurally cannot mint T3.
- The §5 dedup ledger stores **post-extraction T2** (never raw T3). A replay (`IntentReplayComplete`) returns the stored T2 — there are no raw bytes to reconstruct a handle from.
- The shipped C1 `RelayEgressClient.fire(raw_request, ctx, call_index) -> Fired(EgressResponse) | Deduplicated(stored_t2, language)` is ledger-wrapped (commit-then-fire) and audits every refusal path. The shipped C2 `EgressResponseExtractor.handle(raw_request, ctx, call_index, schema, language) -> EgressExtractOutcome` is the §4.3 boundary: on `Fired` it mints a `ContentHandle`, stages the body **in-memory** via `T3BodyRecorder` (transient, popped on extract), runs `quarantined_to_structured`, and records the post-extraction T2; on `Deduplicated` it returns the stored T2 without re-extracting (HARD rule #5).

**Consequence (the shape change):** the re-homed `web.fetch` becomes a **fetch+extract unit** returning a T2 `EgressExtractOutcome`, replacing today's opaque-T3-`ContentHandle` deferred-extraction contract. This is forced by the ledger storing post-extraction T2 and is locked by the launching directive ("the §4.3 extract MUST go through `quarantined_to_structured`"). `dispatch_web_fetch` has **zero production callers** today (verified), so the only blast radius of the return-type change is tests.

## 3. Resolved decisions (specialist panel, 2026-06-28)

A four-lens panel (security / architect / AI-expert / test-engineer) adjudicated three open forks.

### D1 — MIME/content-type enforcement placement: **extend C2 with a pre-extract response-inspection seam** (unanimous `1a`)

The gateway `EgressRelay` enforces method (GET-only), https-pin, the full SSRF chain, DLP stages 2+3 (on request body+URL+header values), TLS origination, redirect-refusal, a 10 MiB response cap, and a wall-clock deadline — but it does **not** enforce MIME, and its allowlist is host:port-only (no path-prefix). MIME is web.fetch *content policy*, not a generic-relay concern, so:

- Add an **optional pre-extract response-inspection seam** to `EgressResponseExtractor`, invoked on the `Fired` branch *after* the relay returns the `EgressResponse` and *before* minting/staging/extracting. `web.fetch` supplies the inspection; other tools supply none (the seam stays generic).
- The seam reads the (attacker-controlled, T3) `EgressResponse.headers`/`.body` and enforces web.fetch's **MIME allowlist** (`text/html`, `text/plain`, `application/json`, `application/xml`, `text/markdown`), **size limit** (web.fetch's own 5 MiB, tighter than the gateway's 10 MiB ceiling), and the **inbound canary scan** (see §4, security folds 4–5).
- **Two outcome classes.** A content-policy mismatch (MIME/size) is a **soft refusal** → recorded as a `TypedRefusal` in the ledger so a §5 replay short-circuits (the fetch already spent its at-most-once budget). An **inbound canary trip** is a **loud refusal** → a non-skippable audit row + raise (HARD rule #7), not a quiet `TypedRefusal`.

**Framing (load-bearing):** `Content-Type` is attacker-controlled T3 — the MIME gate is an *advisory* cost/quality/defense-in-depth narrowing, **never** an injection control. The real containment is the dual-LLM split + Pydantic schema validation, which hold regardless of MIME. Fail closed on a missing/duplicate/garbage `Content-Type` (parse off `;charset=`, case-insensitive).

> This **overrides** the Part D sketch in `docs/superpowers/plans/2026-06-27-g7-2-tool-egress-relay.md:667`, which placed MIME "to `gateway/egress_relay.py`'s fetch path." That line is corrected by this spec.

### D2 — extraction schema source: **required `schema` param on `dispatch_web_fetch`** (unanimous `2a`)

The extraction schema *is* the trust boundary — it bounds what the orchestrator may receive from attacker-influenced T3 bytes — and is inherently **per-call** ("the price as a `Decimal`" vs. "the article's claims" want different typed shapes). #339 (the LLM tool-calling subsystem) owns the live tool-call→schema mapping and the injection-driven-URL corpus. A baked-in default would be a wide free-text passthrough (the weakest boundary shape) and would prejudge #339. C2 already requires `schema`. So `dispatch_web_fetch` takes a **required** `schema: type[ExtractionSchema]`; tests supply a small purpose-built fixture schema; #339 supplies the real one later. No in-seam default; if a generic default is ever wanted it is a *named* schema constant the caller passes, never an implicit fallback.

### D3 — handle-cap fate: **remove it** (`3b`), with the per-user bound deferred to #339 and de-2026-004 re-targeted (3-to-1; security dissented `3a`, reconciled below)

The `HandleCap`'s sole stated rationale (its design doc) is bounding **parked T3 bodies in the Redis content-store** per user. The fused fetch+extract **eliminates that resource**: the body is staged in-memory and extracted synchronously in the same call — there is no parked-handle window. Keeping the Redis-ZSET-TTL machinery around a synchronous unit is mismatched (its reserve/release semantics are keyed to content-store body lifetime).

- **Remove** the `HandleCap` from the web.fetch path (and the host-pre-mint/verify dance C2 makes moot — C2 mints the handle internally; there is no untrusted plugin id to verify).
- **Reconciling the security dissent.** Security correctly noted the C1 semaphore is *global*, not per-user, so per-user fairness is lost. But this has **zero production exposure in G7-2.5**: `web.fetch` has no live caller until #339. The per-user bound becomes load-bearing only when #339 wires a live caller + the real turn-user — which is exactly where the per-(user, persona) quarantine burst-limiter (already gated on #339) belongs. So the per-user fairness/concurrency bound is a **tracked residual gated on #339**, not a silent drop (see §6).
- **de-2026-004** (the release-blocking adversarial test) is **re-targeted, not retired** — re-pointed at the C1 global in-flight HoL bound (anchored alongside `tests/integration/egress/test_quarantine_contention.py`), with the per-user-fairness gap recorded as a residual. The disposition gets **explicit `alfred-security-engineer` sign-off** at PR review.

## 4. Security concerns folded into the design (with disposition)

The panel's security lens surfaced concerns beyond the three forks. Every one is tracked here with where it is addressed. **None is dropped.**

| # | Concern | Severity | Disposition |
|---|---|---|---|
| 1 | Core-side DLP must cover **URL + headers**, not just body (C1 redacts body only; web.fetch auth/secrets ride in headers/URL → unredacted secrets reach the gateway, which then denies or auth fails) | CRITICAL | **G7-2.5 PR1.** `dispatch_web_fetch` runs core-side `OutboundDlp` over the URL + each header before building `_RawToolRequest`. **URL: refuse-on-secret** (cannot send a redacted destination the gateway must fetch). **Headers: redact-and-send** (matches today's per-header redaction). The gateway DLP 2nd pass remains the independent defense-in-depth layer. |
| 2 | The **3-way path-prefix allowlist** + the §7.4 `manifest_broadening_capped` audit would be silently dropped (gateway allowlist is host:port-only → widens `example.com/public/` to the whole host) | HIGH | **G7-2.5 PR1.** Keep the 3-way `AllowlistIntersection` (manifest ∩ operator ∩ session, path-prefix) + the broadening-cap audit core-side in `dispatch_web_fetch`, before fire. This is the J1 "per-session grant narrows within the gateway's structural ceiling" framing made concrete. |
| 3 | No core-side **success audit row** for the T3→T2 crossing (C2 writes only the ledger; the gateway audits the forward; neither carries user/persona/turn attribution + extraction outcome) | HIGH | **G7-2.5 PR1.** `dispatch_web_fetch` emits a payload-blind success audit row (destination host-only, real `EgressResponse.status`, `egress_id`, language, Extracted-vs-TypedRefusal outcome). This is where the new real `status` lands (today hardcoded `None`). |
| 4 | **Inbound canary scan** of the fetched body lost (gateway canary is outbound-only; detecting *our* seeded canary reflected back signals an exfil loop) | MEDIUM | **G7-2.5 PR1** via the C2 pre-extract seam (D1). A trip is a loud audit + raise (HARD rule #7). If wiring proves complex in plan-review, deferral requires explicit security sign-off — not a silent drop. |
| 5 | 10 MiB gateway cap vs. web.fetch's tighter size limit (each MiB now costs an LLM extraction) | MEDIUM | **G7-2.5 PR1** via the C2 pre-extract seam (D1): re-impose web.fetch's 5 MiB limit core-side before extract. |
| 6a | Dedup key omits **schema identity** (replay-with-different-schema returns stale-T2; HARD rule #5 still holds — stored T2, no raw-T3 re-ingest — so staleness, not leak) | LOW | **G7-2.5 PR1.** Assert/document schema stability across the dedup window (same logical call = same schema); fold schema identity into the request shape if cheap. |
| 6b | `status` is attacker-claimed; bytes→text decode must fail closed (a lossy decode could drop a seeded canary before the inbound scan) | LOW | **G7-2.5 PR1.** Validate `status` is a plausible HTTP code; run the inbound canary on **raw bytes** before any lossy decode; pin the decode policy. |

**Correctly-handled (no action):** the gateway's resolve→validate→pin-IP→`sni_hostname` flow closes the DNS-rebinding TOCTOU; `follow_redirects=False` → `UPSTREAM_REDIRECT_REFUSED` is stricter than the old subprocess; `EgressExtractOutcome` omits response headers so T3 headers never reach the orchestrator; web.fetch sets `_RawToolRequest.idempotent=True` (a GET → an in-doubt replay re-fires rather than hard-refusing).

## 5. Design

### 5.1 The re-homed `dispatch_web_fetch` pipeline

`dispatch_web_fetch` keeps its host-side gatekeeping, replaces the subprocess dispatch with the C2 seam, and returns T2:

1. Core-side `OutboundDlp` over **URL** (refuse-on-secret + audit) and **each header** (redact-and-send) — fold #1.
2. **3-way path-prefix allowlist** check (refuse + audit) + the **broadening-cap** audit (once-per-session latch) — fold #2.
3. **Rate-limit** (`RateLimiter.check_and_increment`; per-domain / per-user / daily) — kept (the gateway does not rate-limit).
4. Build `_RawToolRequest(method="GET", url=<real url>, headers=<redacted headers>, body="", idempotent=True)`; thread the explicit `egress_ctx: TurnEgressContext` + `call_index: int` + `schema: type[ExtractionSchema]` params (CORE-3: threaded, never synthesized from `correlation_id`).
5. `await EgressResponseExtractor.handle(raw_request=..., ctx=egress_ctx, call_index=..., schema=..., language=...)` → C1 commits the ledger intent, fires the gateway relay, returns `Fired(bytes)` or `Deduplicated(T2)`. On `Fired`, the **pre-extract seam** (D1) runs MIME/size/inbound-canary; then mint `ContentHandle` → stage T3 → `quarantined_to_structured` → `record_response(T2)`.
6. **Success audit row** with the real `status` + the Extracted-vs-TypedRefusal outcome — fold #3.
7. Return `EgressExtractOutcome` (T2).

**Removed core-side** (the gateway now owns them structurally, and the core goes connectivity-free at G7-3 so it cannot even DNS-resolve): the host-IP SSRF guard, redirect-refusal, the response-size cap *origination* (re-imposed only as a core-side ceiling re-check per fold #5), TLS-policy origination, and the `alfred_web_fetch` subprocess fetch path itself.

### 5.2 The C2 pre-extract response-inspection seam

`EgressResponseExtractor` gains an optional, generic pre-extract inspection invoked on the `Fired` branch before staging/extracting. web.fetch supplies an inspection that enforces MIME + size (soft `TypedRefusal`, recorded so replay short-circuits) and the inbound canary scan (loud audit + raise). Other tools supply none. The exact callable/exception shape and the `TypedRefusalReason` mapping for a MIME/size refusal (map to an existing closed-vocab token vs. add one) are settled in the plan with security review — the closed audit vocabulary is security-owned.

### 5.3 Compose wiring (PR2 — the live cutover)

- `alfred-core`: `ALFRED_EGRESS_RELAY_URL=http://alfred-gateway:8890`.
- `alfred-gateway`: `ALFRED_EGRESS_RELAY_PORT` + `ALFRED_TOOL_EGRESS_ALLOWLIST` + `ALFRED_CANARY_TOKENS` (public, non-secret; relay port **never** under `ports:`).
- `.env.example` documents all four.
- A **positive** compose-invariant test: core's `ALFRED_EGRESS_RELAY_URL` host:port == the gateway's relay port; the canary/allowlist env is present on both; the relay port is not host-published.
- The **production assembly** (core composition root) builds `RelayEgressClient` + `EgressResponseExtractor` from settings/env and hands them to `dispatch_web_fetch`. Safe because nothing live calls `dispatch_web_fetch` until #339 (after G7-3), and the core keeps its direct fallback until G7-3.

## 6. PR decomposition

Two PRs, each leaving `main` coherent (no live LLM caller exists, so neither opens a production-egress window):

- **PR1 — dispatcher re-home + C2 seam + test rewrite (zero live egress).** `dispatch_web_fetch` → fetch+extract via C2 (returns `EgressExtractOutcome`); the C2 pre-extract seam (D1); the kept core machinery (URL/header DLP fold #1, path-prefix allowlist fold #2, rate-limit, success audit fold #3); remove the subprocess fetch path; remove `HandleCap` (D3) + re-target de-2026-004; folds #4–#6. **From-scratch rewrite** of the 59 `test_fetch_dispatcher.py` tests (treat the old suite as a *property inventory* — for each, classify "code deleted → property gone" vs "property survives → must reappear"); ~30–35 delete as dead code (the `ControlResult`/`_ERROR_TYPE_MAP` subprocess-envelope surface, the host-pre-mint/verify tests, the parent-side SSRF tests, the subprocess TLS-threading tests, the `WebFetchDispatchParams` wire-shape tests), the rest rewrite against a loopback relay (reuse `tests/helpers/egress_doubles.py` / `fake_external_world`). A new content-type-laundering adversarial corpus entry (`de-`) driven through the loopback relay. Full `/review-pr` fleet (security ALWAYS) + the adversarial suite (the C2 seam sits on the T3→T2 crossing). Rewrite is gated on the D3 security sign-off (the `HandleCap` deletions are legitimate only because the code is deleted).
- **PR2 — compose + production assembly + live cutover (§5.3).** Gateway relay env wiring, the positive compose-invariant, the core composition-root assembly, `.env.example`. Smoke/devops review.
