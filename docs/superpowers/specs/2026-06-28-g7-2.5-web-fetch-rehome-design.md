# G7-2.5 — The Live `web.fetch` Re-home (Spec C, #333)

**Status:** design — approved 2026-06-28; hardened by a 9-lens `/review-plan` fleet (2026-06-28); implementation pending (writing-plans next).
**Epic:** #333 (Spec C — egress control plane / connectivity-free core).
**Predecessor:** G7-2c (PRs #345/#346, merged `d782812d`) shipped the in-core relay client (C1), the §4.3 gate-checked extractor (C2), the tri-state dedup ledger, and the synthetic-driver proof.
**Successor / hard dependant:** G7-3 (the `internal:true` flip + deletion of the core's direct egress fallback) is **hard-blocked on G7-2.5 landing** — a live tool must not egress while the core's direct fallback still exists.

---

## 1. Goal

Re-home the live `web.fetch` tool egress out of the `alfred_web_fetch` plugin subprocess so the **gateway** performs the fetch via the shipped `EgressRelay`, and `dispatch_web_fetch` reaches the real relay. This is the **side-effecting cutover** that makes the G7-2c relay machinery a production path (its first production consumer). It is L-sized: it moves real enforcement (TLS/size/MIME/redirect) and the content-store/`ContentHandle` mint, not a "re-point."

**This slice does NOT** flip `internal:true`, delete the core's direct egress fallback, add kernel/DNS enforcement tests (all G7-3), harden Discord egress (G7-4), or rewrite PRD §5/§7.1 / the comprehensive ADR-0040 (G7-5, human-gated). It also does **not** add a live LLM caller — `dispatch_web_fetch` remains called only by tests until #339 wires the tool-calling loop after G7-3.

## 2. Locked context (decided in prior slices; not relitigated here)

- The connectivity-free core cannot open external sockets. Tool egress goes core → gateway `EgressRelay` over a length-prefixed JSON-frame wire. The gateway **fetches** the upstream and returns `EgressResponse(status: int, headers: Mapping[str, str], body: bytes)`.
- The **core** mints the T3 `ContentHandle` + tags T3 + runs the **one** sanctioned gate-checked seam `quarantined_to_structured(handle, schema, *, extractor, gate)` (gate-first; a `False` clearance raises `AlfredError`). The gateway holds no `CapabilityGateNonce` (ADR-0036), so it structurally cannot mint T3.
- The §5 dedup ledger stores **post-extraction T2** (never raw T3). A replay (`IntentReplayComplete`) returns the stored T2 — there are no raw bytes to reconstruct a handle from.
- The shipped C1 `RelayEgressClient.fire(raw_request, ctx, call_index) -> Fired(EgressResponse) | Deduplicated(stored_t2, language)` is ledger-wrapped (commit-then-fire) and audits every refusal path. The shipped C2 `EgressResponseExtractor.handle(raw_request, ctx, call_index, schema, language) -> EgressExtractOutcome` is the §4.3 boundary: on `Fired` it mints a `ContentHandle`, stages the body **in-memory** via `T3BodyRecorder` (into an unbounded, no-TTL `QuarantineStagingMap`, drained only when the extractor's transport dispatches), runs `quarantined_to_structured`, and records the post-extraction T2; on `Deduplicated` it returns the stored T2 without re-extracting (HARD rule #5).

**Consequence (the shape change):** the re-homed `web.fetch` becomes a **fetch+extract unit** returning a T2 `EgressExtractOutcome`, replacing today's opaque-T3-`ContentHandle` deferred-extraction contract. This is forced by the ledger storing post-extraction T2 and is locked by the launching directive ("the §4.3 extract MUST go through `quarantined_to_structured`"). `dispatch_web_fetch` has **zero production callers** today (verified), so the only blast radius of the return-type change is tests.

## 3. Resolved decisions (specialist panel 2026-06-28; hardened by plan-review)

### D1 — MIME/content-type enforcement placement: **extend C2 with a pre-extract response-inspection seam** (unanimous `1a`)

The gateway `EgressRelay` enforces method (GET-only), https-pin, the full SSRF chain, DLP stages 2+3 (on request body+URL+header values), TLS origination, redirect-refusal, a 10 MiB response cap, and a wall-clock deadline — but it does **not** enforce MIME, and its allowlist is host:port-only (no path-prefix). MIME is web.fetch *content policy*, not a generic-relay concern, so add an **optional pre-extract response-inspection seam** to `EgressResponseExtractor`, invoked on the `Fired` branch *after* the relay returns the `EgressResponse` and *before* minting/staging/extracting. `web.fetch` supplies the inspection; other tools supply none (the seam stays generic). The seam enforces web.fetch's **MIME allowlist** (`text/html`, `text/plain`, `application/json`, `application/xml`, `text/markdown`), **size limit** (web.fetch's own 5 MiB, tighter than the gateway's 10 MiB ceiling), and the **inbound canary scan** (§4 folds C4/C12).

**Two outcome classes, with their ledger disposition pinned (plan-review TE/SEC/REV):**

- A content-policy mismatch (MIME/size) is a **soft refusal** → recorded as a `TypedRefusal` in the ledger so a §5 replay short-circuits (the fetch already spent its at-most-once budget). It runs **before** staging, so it never stages a body (no leak).
- An **inbound canary trip** is a **loud refusal** → a non-skippable audit row + raise (HARD rule #7). Because a raise leaves the ledger `committed_no_response` and web.fetch is `idempotent=True`, a naive raise would **re-fire on replay and re-send our seeded canary to a hostile origin** — so the canary trip must additionally move the ledger to a **terminal refused** state (no re-fire). Security owns the exact mechanic (§4 fold C8).

**Framing (load-bearing):** `Content-Type` is attacker-controlled T3 — the MIME gate is an *advisory* cost/quality/defense-in-depth narrowing, **never** an injection control. The real containment is the dual-LLM split + Pydantic schema validation, which hold regardless of MIME. Fail closed on a missing/duplicate/garbage `Content-Type` (parse off `;charset=`, case-insensitive).

> This **overrides** the Part D sketch in `docs/superpowers/plans/2026-06-27-g7-2-tool-egress-relay.md:667`, which placed MIME "to `gateway/egress_relay.py`'s fetch path." Size is now enforced at *both* layers (gateway 10 MiB ceiling + core 5 MiB), which that sketch under-describes.

### D2 — extraction schema source: **required `schema` param on `dispatch_web_fetch`** (unanimous `2a`)

The extraction schema *is* the trust boundary — it bounds what the orchestrator may receive from attacker-influenced T3 bytes — and is inherently **per-call**. #339 owns the live tool-call→schema mapping and the injection-driven-URL corpus. A baked-in default would be a wide free-text passthrough (the weakest boundary shape) and would prejudge #339. C2 already requires `schema`. So `dispatch_web_fetch` takes a **required** `schema: type[ExtractionSchema]`; tests supply a small purpose-built fixture schema; #339 supplies the real one later. No in-seam default.

### D3 — handle-cap fate: **remove it** (`3b`), reframed by plan-review

The `HandleCap`'s *stated* rationale (its design doc) is bounding **parked T3 bodies in the Redis content-store** per user. The fused fetch+extract removes the Redis content-store write. **But plan-review (SEC/ARCH) found the parked resource is not eliminated — it moves to the in-memory `QuarantineStagingMap`:** `quarantined_to_structured` is gate-first and raises on a gate denial *before* the extractor's transport drains the staging map, so **every gate-deny / extract-failure orphans a `TaggedContent[T3]` body for process lifetime in an unbounded, no-TTL dict.** Therefore D3's removal is sound **only paired with a C2 drain-on-error fix** (§4 fold C9): stage-after-gate or a `finally` that drains the staging map if extract did not complete. With that fix there is no parked resource and the handle-cap is genuinely vestigial.

- **Reconciling the security dissent.** Security correctly noted the C1 semaphore is *global*, not per-user, so per-user fairness is lost. This has **zero production exposure in G7-2.5** (no live caller until #339); the per-user bound + the per-(user, persona) quarantine burst-limiter belong in #339 when the real turn-user lands. Tracked residual (§7).
- **de-2026-004** (release-blocking adversarial) is **re-targeted, not retired**, but plan-review (TE/SEC) flagged a **property-class downgrade**: the original proves an *active refusal* bound (6th call refused + typed audit row, pre-network); the C1 semaphore *queues, doesn't refuse*. The re-target therefore asserts (a) the global in-flight liveness/no-hang bound (anchored alongside `tests/integration/egress/test_quarantine_contention.py`) **and** (b) **no orphaned T3 body on the gate-deny path** (the drain-on-error fix). The resource-exhaustion-*refusal* payload is explicitly deferred to #339. Requires **explicit `alfred-security-engineer` sign-off** in PR1.

## 5. Design

### 5.1 The re-homed `dispatch_web_fetch` pipeline

`dispatch_web_fetch` keeps its host-side gatekeeping, replaces the subprocess dispatch with the C2 seam, and returns T2. New params threaded explicitly (no global state): `egress_ctx: TurnEgressContext`, `call_index: int`, `schema: type[ExtractionSchema]`, plus the relay/extractor collaborators.

1. Core-side `OutboundDlp` over **URL** (refuse-on-secret + audit; on a DLP-scan *exception*, audit `dlp_scan_error` + a redacted sentinel **before** re-raise — C12) and **each header** (redact-and-send) — C1.
2. **3-way path-prefix allowlist** check (refuse + audit) + the **broadening-cap** audit (once-per-session latch) — C2.
3. **Rate-limit** (`RateLimiter.check_and_increment`) — kept.
4. Build `_RawToolRequest(method="GET", url=<real url>, headers=<redacted headers>, body="", idempotent=True)`; the dedup descriptor (method + URL + schema identity) feeds `compute_body_hash` — C6.
5. `await EgressResponseExtractor.handle(...)` under an explicit **action-deadline** wrapping the fused unit (C13). C1 commits the ledger intent, fires the relay, returns `Fired(bytes)` or `Deduplicated(T2)`. On `Fired` the **pre-extract seam** runs MIME/size (soft `TypedRefusal`) + inbound-canary (loud raise + terminal-refused ledger) on **raw bytes**; then mint `ContentHandle` → stage T3 → `quarantined_to_structured` → (drain-on-error guard, C9) → `record_response(T2)`.
6. **Success audit row** (C3) with the real `status` (C7) + the Extracted-vs-TypedRefusal outcome + language (C11b). A ledger record does not substitute for this row (C12).
7. Return `EgressExtractOutcome` (T2, now carrying `status`).

**Removed core-side** (the gateway owns them structurally; the core goes connectivity-free at G7-3): the host-IP SSRF guard, redirect-refusal, TLS-policy origination, the response-size *origination* (re-imposed only as a 5 MiB core re-check, C5), and the `alfred_web_fetch` subprocess fetch path. The exact module-deletion set (`host_ip_guard`, `tls_policy` origination, the plugin's `_handle_fetch`, web.fetch's `content_store` use) and the plugin-deleted-vs-husk decision are enumerated in the plan with a reviewer-checkable map (§6) **because each removed module that appears in a CI coverage-gate `--include`/`hashFiles()` guard must be reconciled or the gate silently no-ops** (§6).

### 5.2 The C2 pre-extract response-inspection seam

`EgressResponseExtractor` gains: (a) an optional, generic pre-extract inspection invoked on the `Fired` branch *before* staging — web.fetch supplies MIME + size (soft `TypedRefusal`, recorded so replay short-circuits; no gate needed — a refusal is not a T3 dereference, so this is not a tier-laundering hole) + inbound-canary (loud raise + terminal-refused ledger, C8); and (b) a **drain-on-error guard** so a gate-deny / extract-failure does not orphan the staged T3 body (C9). The scan is bounded/non-blocking (C13). The exact `TypedRefusalReason`/audit-`result` token for a MIME/size refusal — map to an existing closed-vocab token vs. add one + migration 0025 — is decided with security in the plan (C10).

### 5.3 Production assembly (PR2 — the live cutover)

- **Compose:** `ALFRED_EGRESS_RELAY_URL=http://alfred-gateway:8890` on `alfred-core`; `ALFRED_EGRESS_RELAY_PORT` + `ALFRED_TOOL_EGRESS_ALLOWLIST` (declare `${...:-}`; unset = default-deny, safe) + `ALFRED_CANARY_TOKENS` on `alfred-gateway` (public, non-secret; relay port **never** under `ports:`); `ALFRED_EGRESS_RELAY_BIND` in `.env.example`.
- **Positive compose-invariant test** (corrected per devops): core's `ALFRED_EGRESS_RELAY_URL` **port == the gateway's `ALFRED_EGRESS_RELAY_PORT`** (mismatch guard); the allowlist/canary env present **on the gateway** (they are gateway-only — the core runs its own 3-way allowlist + core DLP, so they are *not* symmetric like `DEEPSEEK_BASE_URL`); a dedicated "relay port 8890 never host-published" guard (mirroring the 8889 one).
- **The assembly dep graph** (core, per core-engineer): `EgressResponseExtractor` needs a `CapabilityGate` + a `QuarantinedExtractor` + `T3BodyRecorder(nonce=<boot-minted>, staging=<the same map the extractor's transport drains>)` + `OutboundDlp` + a Postgres `EgressIdempotencyStore` + the `RelayEgressClient(relay_url=<settings>)`. **Decision: REUSE the daemon's existing quarantine graph** (the extractor/gate/nonce/staging built at `cli/daemon/_commands.py:~854`) — do **not** spawn a second quarantine child (§4.3 "one production extractor"; CORE-4 shared-child HoL bound). PR2 wires the relay client + ledger + a web.fetch `EgressResponseExtractor` onto that graph.
- **Anti-dead-code:** PR2's assembly has no live caller until #339 (after G7-3) — same **sequencing-enforced** guarantee as G7-2c (G7-3 is hard-blocked on G7-2.5). The integration tests (loopback relay) are the proof, not a live caller.
