# Gateway Adapter-Hosting Inversion (Spec B / G6)

- **Date:** 2026-06-18
- **Status:** Draft (design approved in brainstorming; 11-specialist fleet-reviewed 2026-06-18, all findings folded in; ADR-0036 pending; not yet plan-reviewed). `[fleet]` marks a change that landed from that review.
- **Scope:** This is **Spec B** of the three-epic gateway control-plane program (see [gateway-control-plane-roadmap](2026-06-13-gateway-control-plane-roadmap.md)). Spec B moves comms adapters from daemon-spawned/standalone-service to **gateway-hosted + sandbox-supervised**, so a platform session (Discord) survives a core restart — generalizing the resume [Spec A](2026-06-13-comms-gateway-resume-design.md) proved for the dial-in TUI. **Discord-only**; Telegram (#40) rides the inverted host as a follow-on.
- **Gated on:** Spec A complete and proven in production (#237 graduation criterion #7 closed; G5 / PR #286). **Gates:** Spec C (egress control plane / connectivity-free core) opens only after B proves out.
- **Out of scope (Spec C):** the egress chokepoint for tool/provider I/O, the connectivity-free-core network split, and the full credential-concentration invariant. Adapters keep their existing external network to Discord in B. The PRD "all external I/O via the gateway / connectivity-free core" invariant lands with **Spec C**, not here.
- **Related:** ADR-0031 (TUI socket / Shape B), ADR-0032/0033 (Spec A wire + lifecycle), ADR-0035 (lifecycle-start credentials optional), ADR-0015/0030 (launcher/sandbox: bwrap kind model, fd-3 key delivery, wheel-co-located child), PR #286 (Spec A G5).

## 1. Problem & scope

Today a comms adapter dies with the core. Discord runs as a standalone long-running `alfred-discord` Compose service (reusing the core image, no SETUID, a `secrets.toml` bind-mount); a core restart (self-modification, crash, `alfred daemon stop`, container recycle) does not by itself kill that service, but the adapter is *not* resume-integrated — it has no buffered/replayed path to a restarted core, and the daemon-spawned adapter model (`_build_comms_adapter_wiring` / `_spawn_comms_adapter`) ties an adapter's life to the core boot graph. Spec A introduced an always-up gateway that holds the **dial-in TUI** across a core restart and replays its un-acked input. Spec B **generalizes that resume to platform adapters** by making the gateway their host and supervisor: the gateway spawns the sandboxed adapter child, owns its full lifecycle, and buffers + replays its inbound to the core across a restart.

**In scope:** invert the *hosting model* for the one existing real adapter (Discord). The gateway host-runner is adapter-agnostic, so Telegram (#40, currently greenfield) becomes a follow-on that simply binds to the already-inverted host.

**Out of scope:** building a new adapter; the egress chokepoint, connectivity-free core, and the cred-concentration invariant (all Spec C).

## 2. Locked decisions

These were resolved during brainstorming (2026-06-18). The credential decision (4) was pre-captured before the session; the rest were decided in it. The fleet review refined the *realization* of several (notably 4 → fd-3 delivery).

1. **Discord-only inversion first.** The gateway host-runner is adapter-agnostic; Telegram (#40) is a separate follow-on epic that rides the inverted host. Keeps Spec B about the *inversion mechanism* + the SETUID security reframe, not about building a new adapter.

2. **Full gateway-local lifecycle; core observes.** All five plugin mechanics — handshake, lifecycle, crash detection, restart-backoff, circuit breaker — terminate in the gateway, because the gateway is the adapter child's parent process and supervisor. The **core never commands** an adapter's lifecycle; it **observes** via audited status notifications. **Qualifier** `[fleet rev-002]`: the core does *provide the spawn credential* (decision 4), which is a precondition of a (re)spawn — so "core never commands" means the core issues no start/stop/restart *directives*; it is not a claim that a (re)spawn is possible with the core down (see §5). Cross-wire carries only: T3 inbound bodies (adapter→core), `outbound.message` results (core→adapter), adapter **status** notifications (up/down/crashed/breaker-open), and the spawn-time credential hand-off. This maximizes resume — running adapters survive a core restart untouched.

3. **Gateway gains `cap_add: SETUID` + bubblewrap + the launcher** *(maintainer security co-sign, 2026-06-18).* Once the gateway hosts the adapter runner, *something* in the always-up tier must spawn the bwrap-sandboxed adapter child — and must be able to do so during a core outage, or resume breaks. Decision: that something is the **gateway itself**, a single privileged always-up host, rather than a separate spawner sidecar (same trust tier, added IPC, marginal isolation). The gateway is reframed from "dumb/stable/low-privilege" to **"stable-code, privileged."**
   - **devops-010 reconciliation** `[fleet devops-006, arch-005]`: change the compose-invariant test to a *positive allowed-set* assertion — `services-with-SETUID == {alfred-core, alfred-gateway}` — and **delete** the `alfred-discord` SETUID/state_git assertions (they pass vacuously once the service is removed). Adapters never get SETUID or `state_git`; the gateway never mounts `state_git`.
   - **Concentration acknowledged, bounded by enforcement-of-record** `[fleet sec-004]`: the gateway is now the most network-adjacent process *and* SETUID-privileged *and* the transient credential handler. The honest scope includes that a **compromised gateway can serially harvest every platform credential** via legitimate `spawn_request`/`spawn_grant` calls — this is distinct from (and not mitigated by) "holds no vault key." The bound is per-adapter bwrap + netns + scrubbed env + fd-3 transient cred + no vault key — *not* the process count. Recorded explicitly in ADR-0036; the serial-harvest residual is closed by Spec C's connectivity-free core (the gateway loses the network that makes harvest exfiltrable) and the C-era core→child fd-pass.
   - **Considered and rejected:** rootless bwrap via unprivileged user namespaces (no SETUID) — rejected because AlfredOS deliberately chose the privileged-launcher model for hardened hosts where unprivileged userns is disabled; changing that is a global sandbox-model change, out of Spec B's scope.

4. **Credential = core-injects-at-spawn**, realized in B as **fd-3 transient delivery** `[fleet sec-001]`. The credential-concentration invariant ("no network-holding process decrypts >1 platform cred plaintext") is fundamentally a **Spec C** property — in B the core still has external network *and* holds the vault, so *some* process always has network+all-creds regardless of option. B's job is therefore **correct placement so C flips the switch with zero cred movement**, not satisfying the invariant. Mechanism:
   - On each (re)spawn the gateway sends `gateway.adapter.spawn_request{adapter_id}` to the core over the trusted leg; the core resolves the platform credential from its vault/secret-broker and returns `core.adapter.spawn_grant{adapter_id, credential_material}` (bound to the per-core-boot epoch, dedup-keyed; see §6).
   - The gateway delivers the plaintext to the bwrap child **over fd 3**, mirroring the existing `quarantine_child_io.deliver_*_via_fd3` discipline — **never** via the child's environment. This keeps the credential out of `/proc/<pid>/environ`. The gateway holds the plaintext only transiently (received over the wire, written to the child's fd-3 write-end, then its own copy zeroed); it never retains across spawns, never caches, never holds a vault key, never decrypts.
   - **Why not env** `[fleet sec-001]`: `/proc/<pid>/environ` is readable for the child's whole life by anything in its pid namespace; env injection would also be an unjustified exception to the secret-broker-not-env HARD rule. fd-3 is the established in-tree channel.
   - **B-achievable testable invariant:** *the gateway never holds a vault key and never retains a credential past a single spawn; the credential never appears in the child's environment.*
   - **Deferred to C** `[fleet arch-004]`: the core writes the credential **directly to the child's fd, bypassing the gateway entirely** (SCM_RIGHTS fd-pass), so the gateway never transits plaintext at all — closing the serial-harvest residual of decision 3. B's spawn contract is shaped so this is a drop-in swap (same fd-3 child-read path; only the *writer* moves from gateway to core).

5. **Gateway-ingress gate = coarse two-tier per-adapter, payload-blind.** The gateway enforces a per-**adapter** sustained-rate token bucket **plus** an in-flight/concurrency cap on the adapter→core leg — bounding any single adapter's flood before gateway buffering + the wire, *without* parsing identities. Per-platform-id fairness stays the core's existing `_PreResolutionLimiter` (where identity is authoritatively resolved), demoted to defense-in-depth and **explicitly not weakened or consolidated away** `[fleet sec-005]` — the gateway cap is *additive*. **This is an explicit overturn of roadmap finding comms-F2** `[fleet rev-003]` (which proposed a per-platform-id gate at the gateway); rationale: payload-blindness is a load-bearing invariant; a flood is bounded correctly by *volume*, not identity (per-id keying misses the distributed-id case and would leak a stable per-user identifier into the most-exposed process); defense-in-depth means *different* checks per layer, not a weaker duplicate of the core's. On trip: `ReplayBuffer` back-pressure + `gateway_ingress_throttled_total{adapter}` metric + audit row; never a silent drop.

6. **Clean flag-day, no dual-mode.** Mirror Spec A's G5 discipline (which deleted the #259 direct-dial path outright). One epic removes the `alfred-discord` Compose service, moves its secret handling to the core-injects-at-spawn path, deletes the `docker compose run --rm alfred-discord verify` flow, and migrates the full Discord test suite. No window where both hosting models exist. The cutover ships with an **operator migration runbook** (§7) because a `git pull` is otherwise a silent breaking change for existing deployments.

## 3. Topology & data flow

```
 Discord API ◀──▶ [bwrap adapter child]  ┐
                       (gateway-spawned,  │  adapter_id-tagged, payload-blind frames
                        SETUID launcher,  ├──▶ alfred-gateway ──▶ alfred-core (daemon)
                        cred over fd-3,   │   (relay + per-leg      (identity resolve,
                        scrubbed env)     │    ReplayBuffer +        _PreResolutionLimiter,
 alfred chat ──▶ [TUI client] ───────────┘    adapter supervisor +  G0 idempotency commit,
   (Spec A dial-in, unchanged)                 ingress gate +        T3 extract, dispatch,
                                               fair per-leg sched)   outbound wire-sender)
```

- The gateway becomes a **multiplexer** over two *distinct* host mechanics `[fleet comms-005]`: the TUI dial-**in** leg (Spec A — the client dials, the gateway *accepts* a connection) **plus** N gateway-**spawned** adapter legs (the gateway forks/execs the child). Only the spawned/pipe carrier is *inverted* by Spec B; the TUI accept-carrier is unchanged. All legs multiplex to the core over the existing `0600` / `SO_PEERCRED` / per-boot-epoch gateway↔core link.
- `adapter_id` rides the **out-of-band envelope** — the gateway knows it because it *spawned* the child, so it is **not payload-derived** and payload-blindness holds. The core already keys its G0 idempotency commit on `(adapter_id, inbound_id)`, so the relayed frame re-enters the existing core inbound pipeline cleanly.
- **Inbound:** adapter child → gateway (host runner) → core. Each adapter leg gets its **own `ReplayBuffer`** (Spec A's per-direction buffer, instantiated per leg) → platform inbound survives a core restart, replayed FIFO, deduped by the core's G0 commit-before-side-effect. **The relayed inbound frame re-enters the exact existing core graph** `[fleet comms-003]`: the daemon's `_CommsBootGraph` (per-adapter `SubPayloadPromoter` + `ContentStore` + the bwrap quarantine transport + session/handlers) stays core-side and `process_inbound_message` runs unchanged.
- **Outbound:** core `outbound.message` → gateway → adapter child. **This is a trust-boundary split of a currently in-process binding** `[fleet comms-001]`: today `_build_comms_runner` binds the core orchestrator's outbound sender directly to the runner (`bind_outbound_sender(_RunnerOutboundSender(runner))`). With the runner relocated to the gateway, the core's outbound sender becomes a **core→gateway wire sender**; the gateway's runner re-issues `send_request('outbound.message')` to the child. **Note** `[fleet rev-005]`: there is no real persona-outbound reply path yet (dispatch emits a fixed ack; #235); B exercises the *wire-sender plumbing* against that fixed-ack path, and real persona-outbound remains a separate epic.
- **Status:** gateway → core id-less notifications `gateway.adapter.{up,down,crashed,breaker_open}`, **Pydantic-validated core-side** (G3 anti-forgery lesson: never trust a raw frame; a forged "up" must be refused — see §6) → audit + `alfred status` + metrics. The core consumes status; it does not issue lifecycle commands. **Two crash signals coexist** `[fleet comms-004]`: the in-child `crash_emitter` ("the adapter's own code hit an error") and the gateway's `gateway.adapter.crashed` ("the child process exited"). The gateway's process-level signal is **authoritative for host-supervision/audit**; the in-child emitter is a finer-grained diagnostic. The core de-dups by correlating both on `adapter_id` + a host-restart sequence so a single crash is one audited incident.

## 4. Components

**Gateway-side (new, `src/alfred/gateway/`):**

- **`GatewayAdapterSupervisor`** — owns the per-adapter lifecycle: spawn via the launcher, handshake, crash detection, bounded-backoff restart, per-adapter circuit breaker, status emission. Mirrors the daemon plugin supervisor's loudness contract. Reuses `CommsPluginRunner` + `CommsStdioTransport` (the existing host-side runner/transport), relocated/shared into the gateway host role. **Boot + recovery spawn concurrency** `[fleet perf-004]`: multi-adapter boot spawns concurrently under a bounded `TaskGroup` (not serial), and a multi-adapter restart-storm on core-return is avoided by **per-adapter decorrelated backoff jitter** so N adapters do not stampede `spawn_request` simultaneously.
- **`GatewayAdapterCredentialClient`** — the spawn-time `spawn_request → spawn_grant` round-trip; receives plaintext over the wire, writes it to the child's fd-3, zeroes its own copy after the write; no retention. Fail-closed (§5/§6). Blocks (with bounded backoff + a terminal alert ceiling, §5) on the core link when the core is down.
- **`PerAdapterIngressGate`** — two-tier token-bucket (sustained rate) + in-flight/concurrency cap; wired to `ReplayBuffer` back-pressure + metrics + audit. Releases its in-flight slot on TTL-evict so a stalled leg cannot wedge `[fleet perf-003]`.
- **`GatewayLegScheduler`** `[fleet perf-001]` — fair scheduling across the N adapter legs + the TUI leg over the single-writer core link, with a bounded per-leg send queue, so a chatty/large-payload adapter cannot serialize ahead of live `alfred chat` or starve another adapter. Egress to the core runs as isolated per-leg tasks.
- **`ReplayBuffer`** (Spec A, **reused**) — instantiated per hosted-adapter leg in addition to the TUI leg, under a **global aggregate cap** across all legs `[fleet perf-002]` (not just per-leg) so total resident pre-DLP T1 in the always-up SETUID process is bounded regardless of N.

**Core-side (new + unchanged):**

- **`CoreAdapterCredentialResolver`** (new) — resolves a vault/secret-broker credential → `spawn_grant` over the trusted leg; the only component that decrypts the platform cred.
- **Core→gateway outbound wire-sender** (new) `[fleet comms-001]` — replaces the in-process `_RunnerOutboundSender` binding; forwards `outbound.message` over the wire to the gateway's runner.
- **Adapter-status observer/auditor** (new) — consumes + Pydantic-validates `gateway.adapter.*` notifications → audit rows + status surface + metrics; refuses forged/malformed frames loudly (§6).
- **Unchanged, reached over the wire:** `process_inbound_message`, `_PreResolutionLimiter`, identity resolve (`_IdentityResolverLike`), the G0 `InboundIdempotencyCommit`, sub-payload promotion, the quarantined extractor. Spec B does **not** touch the core's inbound semantics — only *where the adapter that feeds them is hosted*.

**Adapter-plugin-side (changed)** `[fleet comms-002]`: the Discord adapter currently fetches its own token inside the child (`DiscordLifecycle.start → self._broker.get('discord_bot_token')`). Under core-injects-at-spawn the child must **stop self-brokering its token** and instead read it from fd-3 at startup. This is a real (small) plugin change, owned by G6-5, beyond the Compose/secret relocation.

**Injectable seams** (test): fake clock (reuse Spec A's), explicit adapter-lifecycle state machine, a **fake-core for `spawn_grant`/`spawn_request`** (so the lifecycle machine is testable before the real cred client lands), a fake adapter child, a fake fd-3 sink.

## 5. Credential-during-core-outage (resilience vs. retention)

Decision 2 makes restart gateway-local; decision 4 makes every spawn require a fresh core-provided credential. These meet at the question: *when an adapter crashes while the core is down, where does the re-spawn credential come from?*

**Default (recommended): every (re)spawn requires a fresh `spawn_grant`; the gateway never caches the credential.**

- **Running adapters survive a core restart untouched** — they hold their connection and do not need re-credentialing — so the headline resume value is fully preserved.
- The **only** gap is an adapter that *crashes during* a core-down window: its restart loop **waits for the core link to return**, then spawns once the core issues a fresh grant. A narrow double-fault.
- This keeps the strongest credential posture: **zero gateway retention** beyond a single spawn instant.

**Loudness + bounds** `[fleet sec-003, err-001, err-002, test-006]`: the await-core wait is **non-spin** (bounded backoff with decorrelated jitter), **loud + audited** (`adapter awaiting core for re-spawn`), and has a **terminal alert ceiling** — past it the adapter is marked durably down with a distinct alert (no quiet-dark). The wait interacts with the per-adapter breaker (the breaker does not silently mask the awaiting-core state). A `spawn_request` interrupted mid-round-trip by a link drop surfaces a **typed loud error** (mirrors Spec A's gateway-unreachable handling), not a hang.

**Fail-closed credential pipeline** `[fleet err-001]`: `spawn_grant` refusal, launcher-spawn failure, and **fd-3 write / zero-after-write failure** each raise a loud, audited `QuarantineChildSpawnError`-analog and abort the spawn — never log-and-continue. A credential that cannot be confirmed zeroed fails the spawn.

**Overturn option (rejected by default):** let the gateway cache the resolved credential in `MADV_DONTDUMP` / zeroed memory so it can re-spawn a crashed adapter during a core outage. This buys crash-resilience across outages but trades away the no-retention invariant — the gateway would hold platform plaintext over time, in the most network-exposed process. Recommended **against**; revisit only if crash-during-core-outage proves operationally common.

## 6. Trust-boundary posture & honest scoping

- The gateway stays **payload-blind for T3 message bodies**: it parses only the out-of-band envelope (seq/ack/dedup/`adapter_id`); T3 tagging + extraction stay core-side at `process_inbound_message` (hard rule #5 intact — the privileged orchestrator never sees raw T3). The per-leg `ReplayBuffer` + relay treat T3 bodies as **opaque** — no parse, no log `[fleet sec-006]` — so hard rule #5 cannot regress invisibly.
- **Honest wrinkle (recorded in ADR-0036):** with core-injects-at-spawn *where the gateway is the spawner*, the gateway **does** transit *plaintext platform credentials* at spawn time — over the trusted `0600` / `SO_PEERCRED` / epoch leg, transient, written to fd-3, zeroed. "Payload-blind" applies to **message bodies**, not to spawn-time credential control frames. A **compromised gateway can serially harvest credentials** via legitimate grants (decision 3). The fd-pass (core → child, bypassing the gateway) + connectivity-free core are the **Spec C** hardenings that close this; B does not claim the gateway is credential-blind.
- **Per-adapter isolation:** each adapter is a separate bwrap child receiving its credential over its own fd-3 with a scrubbed env and **no shared credential memory** between adapters; a compromise of one adapter cannot read another's credential (no shared env, no shared fd).
- **Buffered pre-DLP input** (per leg) keeps Spec A's bound: cap + TTL + zero-on-trim/breaker + `MADV_DONTDUMP`, under the global aggregate cap (§4).
- **Audit non-skippable:** every adapter lifecycle transition (spawn, handshake, crash, each restart attempt, breaker trip, `awaiting-core-for-respawn`, ingress-gate trip) writes an audit row; status notifications reconcile into the signed core audit log (the gateway holds no signing key, per Spec A). The gateway-local audit append + reconcile (Spec A's mechanism) is **reused for the new credential + awaiting-core control frames** `[fleet arch-002]`, and **reconcile-failure / gateway-local-buffer-overflow fails loud** `[fleet err-003]` (per-adapter volume makes overflow plausible). A malformed/forged status frame is **never silently dropped** `[fleet err-004]` — it is refused, audited, and triggers link scrutiny (Spec A's never-ack-and-drop posture).

**Adversarial corpus (before ship)** `[fleet sec-002, test-001]`:

- **(a)** cross-adapter credential read attempt (one adapter reads another's fd-3 / env) → refused + audited.
- **(b)** `[fleet test-002]` gateway memory inspected after a spawn → **no retained plaintext credential** (mechanically checkable: the credential `bytearray` is zeroed post-write) **and a structural "gateway holds no vault key" invariant**; the `/proc`/core-dump negative is run on the privileged CI lane (or explicitly deferred with rationale).
- **(c)** adapter crash-loop → bounded backoff + breaker + loud audit; never silently dark.
- **(d)** per-adapter ingress flood → bounded + back-pressured + audited (and the **distributed-id** variant still bounded by the volumetric cap).
- **(e) NEW — spoofed / replayed `spawn_grant`** (forged or stale-epoch credential frame) → refused; epoch-reconcile on the cred leg.
- **(f) NEW — forged `gateway.adapter.*` status** (a peer asserting a false "up" while dark) → refused by Pydantic validation + epoch/peer-auth; audited.
- **(g) NEW — compromised-gateway serial cred harvest** → bounded/observable (documents the decision-3 residual; the test asserts each grant is per-adapter + audited so harvest is *visible*, not silent).

**Release-blocking pair → quartet** `[fleet test-001]`: (b) no-retained-cred, (c) bounded-crash-loop, **(e) spawn-grant authenticity, (f) status-forgery refusal** are release-blocking; (a), (d), (g) ship alongside.

## 7. Migration / flag-day (clean cutover, no dual-mode)

**Current-state correction** `[fleet devops-001, rev-004]`: on `main` (75ea76c5) there is **no `alfred-gateway` Compose service**, `alfred-core` is a **one-shot command runner** (not a long-running daemon), and there is **no `ops/` tree** (no `ops/grafana`, no `ops/alerts`). Spec A's G3-4 deployment surface was **deferred/never shipped**. Spec B therefore opens with a **deployment precursor** (G6-0, §9) that lands that surface before any hosting work.

- **Compose:** add the long-running `alfred-gateway` service (`restart: unless-stopped`, shared `alfred_run` volume, the two-tier healthcheck from Spec A §7) with bubblewrap + the launcher + `cap_add: SETUID` in its image; convert `alfred-core` to the long-running daemon service; **delete** the `alfred-discord` service.
- **Secrets:** move Discord credential handling from the standalone-service `secrets.toml` bind-mount to the **core vault → `spawn_grant` → fd-3** path. The new credential source must be a concrete core-side mount/broker entry `[fleet devops-004]` — `alfred-core` mounts no `secrets.toml` today, so this is a new core mount/broker registration + setup-script re-point, not just a deletion. No platform secret is bind-mounted into the gateway.
- **Setup script (`bin/alfred-setup.sh`):** delete the `docker compose run --rm alfred-discord verify` step; replace with a **concrete** `alfred gateway` / adapter-status verification `[fleet devex-002]` — a named command (e.g. `alfred gateway adapters --wait-ready discord`) with a defined **exit-code contract** (mirroring the deleted Discord verify's 0/1/2/3 remediation-naming) that confirms the gateway spawned the Discord adapter and it reached `up`.
- **Operator observability replacement** `[fleet devex-003]`: the adapter becomes a bwrap child *inside* the gateway, so `docker compose logs alfred-discord` returns nothing. Provide a replacement — adapter stdout/stderr surfaced via `alfred gateway logs <adapter>` (or routed to the gateway's structured log with an `adapter_id` field) — so an operator can diagnose a Discord adapter that won't reach `up`.
- **Operator migration runbook** `[fleet devex-001 — CRITICAL]`: a clean flag-day is a **breaking change** for existing deployments. Ship a runbook (in the comms subsystem doc + setup output) enumerating: (1) relocate the Discord token from `secrets.toml` to the core vault, (2) `docker compose up -d` brings up `alfred-gateway` + the core daemon and `alfred-discord` is gone (expected), (3) the new verify command, (4) the new log-access command. Setup must detect the legacy `alfred-discord` config and guide the operator, not fail opaquely.
- **Observability — per-adapter metrics + operator presentation** `[fleet perf-005, devops-002, devex-005]`: G6-0 creates the `ops/` tree; G6-2/G6-4 populate it with **per-adapter, `{adapter}`-labelled** series — `gateway_adapter_up`, `gateway_adapter_restarts_total`, `gateway_adapter_breaker_open`, `gateway_adapter_buffer_depth_{frames,bytes}`, `gateway_adapter_inflight`, `gateway_adapter_awaiting_core`, plus the existing `gateway_ingress_throttled_total` — so saturation/breaker/await-core is attributable to a specific leg (Spec A's gateway metrics are global/unlabelled). The new operator-facing states (`awaiting-core`, `throttled`, `breaker-open`, `crashed`) are **surfaced human-readably** in `alfred gateway adapters` / `alfred status`, each with a documented **sensible default** (ingress rate/burst, breaker threshold + cooldown, await-core terminal ceiling) so an operator can see *why* an adapter is not `up`.
- **Network egress (B keeps it as-is)** `[fleet devops-005]`: adapters retain external network to Discord — there is **no network split in B** (that is Spec C). The gateway keeps default egress; per-adapter network isolation (netns) is a **launcher / sandbox-kind property**, not a compose-network change, so B touches no custom Compose networks.
- **Tests — migrate, do not drop** `[fleet test-004]`: enumerate **all** Discord test files (≈15 unit in `tests/unit/plugins/alfred_discord/` + ≈5 integration + adversarial + smoke) and assign each in the G6-5 plan to *survives-unchanged* (wire-contract: message surface, addressing, thread continuity, sub-payload promotion), *re-pointed-to-gateway-harness* (lifecycle, crash, handshake, server-dispatch), or *changed-by-the-new-auth-model* (the capability-grant model moves from ADR-0027 daemon LOAD-grant → gateway-spawn + core-injected-cred; capability-grant/refusal tests are **rewritten to the new model, not retired** — no coverage loss).
- **Compose-invariant test** (`tests/unit/test_compose_invariants.py`, devops-010): positive allowed-SETUID-set `== {alfred-core, alfred-gateway}`; add `gateway-has-no-state_git`; **delete** the now-vacuous `alfred-discord` assertions `[fleet devops-006]`.
- **Docs:** README quickstart + `docs/subsystems/comms.md` updated for the gateway-hosted model; **add** the human-gated CLAUDE.md command-table `alfred gateway start` / adapter rows (no such row exists today — it must be added, not assumed) `[fleet docs-003, devex-004]`; glossary sweep for new vocabulary. These doc updates land in **G6-5** (flag-day) `[fleet docs-004]`.

## 10. Testing

- **Unit:** adapter-lifecycle state machine (fake clock — spawn/handshake/crash/restart/breaker as pure transitions; non-spin backoff with decorrelated jitter); `PerAdapterIngressGate` (token-bucket + in-flight cap; trip → back-pressure; in-flight slot released on TTL-evict; distributed-id still bounded); `GatewayLegScheduler` (no leg starves another / the TUI); `GatewayAdapterCredentialClient` (fd-3 write + zero-after-write asserted; fail-closed on grant-refusal/launcher-fail/zero-fail; await-core-on-outage non-spin + loud); status-notification encode/validate + forgery refusal.
- **Integration:** gateway spawns the real Discord adapter (mock Discord client) over the launcher; a **core restart at a deterministic barrier** → the Discord session is held and inbound replayed **exactly once** (release-blocking; the per-leg analog of Spec A's restart-survival test). The determinism seam is explicit `[fleet test-003]`: a two-phase assertion — replay **refused** while the core is pre-`ready` (half-booted), **accepted** once `ready` — mirroring Spec A's Task-3 pattern, not a sleep.
- **Adversarial:** the seven §6 corpus entries; (b) no-retained-cred, (c) bounded-crash-loop, (e) spawn-grant authenticity, (f) status-forgery refusal are release-blocking.
- **CI lane** `[fleet test-005, devops-003]`: real-bwrap spawn tests run on the existing required `integration-privileged` (root) job; because root-only tests **skip** on the non-root gate (the #243/#245 paper-gate hazard), every wire-contract/property they prove also gets an **in-process non-root analog** so the required non-root gate is not green-because-skipped.
- **Coverage bar:** the credential fd-3 path + the spawn/cred control-frame handling held to **100% branch** (trust boundary; confirm with security in ADR-0036); ≥80% on the supervisor/ingress/scheduler core; happy/error/refusal triad per component.
- **Assertion discipline** `[fleet i18n-003]`: tests assert operator-facing output via `t()` catalog keys / canonical status IDs, never raw English wording, so a catalog change doesn't break tests and the i18n contract stays enforced.
