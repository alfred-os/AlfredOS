# Slice 2.5 — Pluggable Hooks for Every Action: Design

> **Status:** Draft for review
> **Date:** 2026-05-27
> **Slice:** 2.5 (between Slice 2 graduation and Slice 3's MCP-transport rewrite)
> **Implements:** [ADR-0014](../../adr/0014-pluggable-hooks-for-every-action.md) (with a clarifying amendment, see §9)
> **Forward-references:** ADR-0013 (defer T1/T3 + dual-LLM to Slice 3), ADR-0009 (CommsAdapter Protocol Slice-2-only → Slice-3 MCP)

## 0. Summary

AlfredOS gains a first-class **hookpoint** primitive: a named, string-keyed
extension point that any code — core lifecycle methods, in-tree plugins, or
(from Slice 3) out-of-process MCP plugins — can both **declare** (publish) and
**subscribe to**. Subscribers run in a deterministic, capability-gated chain
and may mutate the action's input, refuse the action, observe or mutate its
result, react to its errors, or run cleanup on cancellation.

This slice ships the **contract and registry**, instruments **one
proof-of-concept action** (`audit.append`, with three internal hookpoints) to
exercise the lifecycle-stage shape end-to-end, and lands a release-blocking
**performance gate**. The real capability-gate implementation and the MCP
transport for out-of-process plugins are explicitly **Slice 3**; this slice
ships against a dev-only `AllowAllGate` because no third-party plugins exist
yet (the MCP transport that would create them is Slice 3).

**No behaviour change to existing actions** beyond `audit.append` gaining
internal hookpoints whose chains are empty by default (zero registered
subscribers → a single dict lookup, no observable change).

## 1. Motivation

ADR-0014 establishes *why* hooks are needed: AlfredOS's household-OS vision
requires third parties (plugins, the operator, system components) to observe or
intercept actions taken by other components without those components knowing
about the third party. The event bus (PRD §5) is observe-only and out-of-band;
the capability gate (PRD §7.1) gates tool *invocation* permission, not
mid-flight interception; the audit log (PRD §7.4) is after-the-fact. None of
these is an in-band interception seam.

This spec turns ADR-0014's architectural commitment into an implementable
contract. The key conceptual move, refined during brainstorming beyond
ADR-0014's original wording: **hookpoints are a general extensibility
primitive, not an outer wrapper around a fixed core-defined action set.** Core
lifecycle methods are themselves hookpoint *producers* — they use the exact
same `hooks.invoke()` primitive a plugin uses to declare its own lifecycle.
There is no asymmetry between "core actions get hooks" and "plugins publish
hooks." This is the pytest-pluggy / Django-signals / WordPress-action+filter
family of designs.

## 2. Glossary

- **Hookpoint** — a named (`"audit.append.before_db_write"`), string-keyed
  extension point declared at a specific point in some code's execution. The
  unit of extensibility.
- **Publisher** — code that declares a hookpoint by calling
  `await hooks.invoke(name, ctx, kind=...)` at a point in its execution. Core
  lifecycle methods and plugins are both publishers.
- **Subscriber** — a function registered against a hookpoint via
  `@hook(name, kind=..., tier=...)`. Runs when the publisher invokes that
  hookpoint.
- **Kind** — one of `pre` / `post` / `error` / `cancel`. Routing semantics for
  a subscriber within a hookpoint invocation (see §4).
- **Tier** — one of `system` / `operator` / `user-plugin`. Determines chain
  ordering and is a capability the operator grants per plugin (see §6).
- **`HookContext[T]`** — the frozen, typed payload threaded through a chain.
- **`HookRefusal`** — the exception a `pre` subscriber raises to short-circuit
  the action.
- **Chain** — the ordered set of subscribers that run for one
  `(hookpoint, kind)` invocation, sorted by tier then registration order.
- **Correlation ID** — a UUID minted by the publisher, shared across every
  hook invocation in one logical action so the chain is reproducible from the
  audit log.

## 3. Architecture

A new top-level subpackage `src/alfred/hooks/`:

```
src/alfred/hooks/
├── __init__.py         # public exports
├── registry.py         # HookRegistry: register subscribers, lookup, dispatch chain
├── context.py          # HookContext[T] frozen generic dataclass
├── errors.py           # HookError → HookRefusal taxonomy (rooted at AlfredError)
├── capability.py       # CapabilityGate Protocol + AllowAllGate impl (dev-only)
├── decorators.py       # @hook(name, kind, tier) subscriber-side decorator
└── invoke.py           # the publisher-side hooks.invoke(name, ctx, kind=...) entrypoint
```

### 3.1 Public surface (the contract everyone codes against)

```python
from alfred.hooks import (
    hook,            # @hook(name, kind, tier) — subscriber decorator
    invoke,          # await invoke(name, ctx, kind=..., subscribable_tiers=...) — publisher
    HookContext,     # HookContext[T] frozen payload
    HookRefusal,     # raised by a pre-hook to short-circuit
    HookError,       # base of the hook exception taxonomy
    CapabilityGate,  # Protocol the registry consults at registration
    AllowAllGate,    # dev-only impl; Slice 3 ships the real gate
)
```

### 3.2 The registry

`HookRegistry` is a per-process singleton (`get_registry()` accessor). The
singleton fits the in-tree decorator pattern, which registers at module-import
time. Tests reset it via a `reset_registry()` fixture so registrations don't
leak between tests.

Responsibilities:
- `register(subscriber)` — validate (async-only, gate-approved tier, hookpoint
  accepts the requested tier), store keyed by `(hookpoint, kind)`, sorted by
  `(tier_rank, registration_seq)`.
- `subscribers_for(hookpoint, kind) -> tuple[Subscriber, ...]` — O(1) dict
  lookup returning the ordered, frozen tuple. Empty tuple is the default.
- `reset()` — test-only.

The registry holds **no per-invocation state**. Invocation state (the
correlation-ID re-entry stack) lives in a `ContextVar` so it is per-task,
matching AlfredOS's existing per-coroutine `_active_lang` pattern.

### 3.3 HookContext

```python
@dataclass(frozen=True, slots=True)
class HookContext[T]:
    action_id: str               # logical action, e.g. "audit.append"
    hookpoint: str               # specific point, e.g. "audit.append.before_db_write"
    input: T                     # the typed action input
    correlation_id: str          # UUID shared across the whole chain
    kind: Literal["pre", "post", "error", "cancel"]
    metadata: Mapping[str, object] = field(default_factory=dict)

    def with_input(self, new_input: T) -> Self:
        return replace(self, input=new_input)

    def with_metadata(self, **kv: object) -> Self:
        return replace(self, metadata={**self.metadata, **kv})
```

Frozen + `with_*` copy methods: subscribers cannot mutate shared state through
the context. (Arbitrary Python side effects inside a subscriber body are
irreducible — the operator trusts plugin code at install time; see §6.)

`T` is a per-action input dataclass. Convention: every hookpoint-publishing
action ships an `<ActionName>Input` frozen dataclass alongside its
implementation. For the PoC, `AuditAppendInput` lives in `src/alfred/audit/log.py`.

### 3.4 Publisher side — `invoke`

```python
async def invoke[T](
    name: str,
    ctx: HookContext[T],
    *,
    kind: Literal["pre", "post", "error", "cancel"],
    subscribable_tiers: frozenset[str] = frozenset({"system", "operator", "user-plugin"}),
    refusable_tiers: frozenset[str] = frozenset({"system", "operator", "user-plugin"}),
    exc: BaseException | None = None,
) -> HookContext[T]:
    ...
```

- `subscribable_tiers` — which tiers MAY have subscribers on this hookpoint.
  Enforced at registration time (a registration for a disallowed tier is
  refused), re-checked at invoke time defensively.
- `refusable_tiers` — which tiers MAY raise `HookRefusal` on this hookpoint.
  A refusal from a tier not in this set is converted to a fail-loud
  `HookError` (the refusal was unauthorized) + audited.
- `exc` — required when `kind="error"`; the exception the action raised.
- Returns the (possibly mutated) `HookContext`. For `pre` the caller uses
  `result.input`; for `post` the caller uses `result.input` as the
  possibly-annotated result carrier.

`invoke` is authoritative for `kind` and `hookpoint`: it rewrites the incoming
context's `kind` and `hookpoint` fields to match its own arguments before
dispatching, so a single context object threaded through multiple stages
always reflects the stage it is currently in. The publisher does not need to
re-set those fields between `invoke` calls.

### 3.5 Subscriber side — `@hook`

```python
@hook("audit.append.before_db_write", kind="pre", tier="system")
async def redact_secrets(ctx: HookContext[AuditAppendInput]) -> HookContext[AuditAppendInput]:
    return ctx.with_input(ctx.input.with_subject(_scrub(ctx.input.subject)))

@hook("audit.append.write_failed", kind="error", tier="system")
async def alert_on_audit_failure(
    ctx: HookContext[AuditAppendInput], exc: Exception
) -> None:
    _send_alert(ctx.correlation_id, exc)
```

- All subscribers are `async def`. A sync function passed to `@hook` is
  rejected at decoration time with `HookError` (fail-loud; AlfredOS core is
  async-first and a sync hook would force a thread-pool hop that breaks the
  in-band guarantee).
- `pre` subscribers return a `HookContext` (mutated or pass-through) or raise
  `HookRefusal`.
- `post` subscribers return a `HookContext` (may annotate `metadata` or mutate
  `input` for downstream observers) or pass through.
- `error` subscribers take `(ctx, exc)` and return a replacement
  `HookContext` to swallow-and-substitute, or re-raise (default).
- `cancel` subscribers take `ctx`, run cleanup only, cannot suppress the
  `CancelledError`.

### 3.6 Plugins as hookpoint publishers

A plugin exposes its own lifecycle by calling `invoke()` from its own code,
exactly as core does. Its manifest lists `hookpoints_published: [...]` for
discovery so other plugins can find and subscribe to them. There is no
registry-side distinction between a core-published and a plugin-published
hookpoint — both are string keys.

## 4. The four kinds and the dispatch sequence

A publishing action threads its lifecycle through named hookpoints. Each
`invoke` call specifies one `kind`. A typical action:

```python
async def append(self, *, event: str, subject: Mapping[str, object], ...) -> None:
    ctx = HookContext(
        action_id="audit.append",
        hookpoint="audit.append.before_validate",
        input=AuditAppendInput(event=event, subject=subject, ...),
        correlation_id=str(uuid4()),
        kind="pre",
    )

    # Stage 1: validate (pre — input mutation allowed)
    ctx = await invoke("audit.append.before_validate", ctx, kind="pre")
    _validate(ctx.input)

    # Stage 2: db write (pre — system/operator may mutate or refuse)
    ctx = await invoke(
        "audit.append.before_db_write", ctx, kind="pre",
        subscribable_tiers=frozenset({"system", "operator"}),
        refusable_tiers=frozenset({"system"}),
    )
    try:
        await self._db.execute(_to_row(ctx.input))
    except Exception as exc:
        # Stage 3a: error (observers + optional swallow-replace)
        await invoke("audit.append.write_failed", ctx, kind="error", exc=exc)
        raise

    # Stage 3b: committed (post — observers)
    await invoke("audit.append.committed", ctx, kind="post")
```

**`pre` chain semantics:** subscribers run in tier order. Each receives the
prior subscriber's outgoing context. The first `HookRefusal` short-circuits:
the action does not run, no further `pre` subscribers run, the refusal is
audited, and `HookRefusal` propagates to the caller. The publisher decides
whether a hookpoint is refusable and by which tiers.

**`post` chain semantics:** runs after the action succeeds. Mutations are for
downstream chain observers; the side effect (DB write) has already happened.

**`error` chain semantics:** runs when the action raises. A subscriber may
return a replacement context to suppress-and-substitute; the default is
re-raise. Order: tier order; first subscriber to return (not re-raise) wins.

**`cancel` chain semantics:** runs on `asyncio.CancelledError`. Cleanup only;
cannot suppress the cancellation.

## 5. Performance

Per ADR-0014, hook dispatch is on hot paths and must be near-free when no
subscribers are registered.

**Budget (release-blocking via `pytest-benchmark`):**
- Empty hookpoint (zero subscribers): **p99 < 10µs** per `invoke`.
- 5-subscriber `pre` chain: **p99 < 100µs** per `invoke`.
- Refusal short-circuit: subscribers after the refusing one do not run
  (correctness assertion paired with the bench).

**Implementation guarantees:**
- The no-subscriber path is a single dict lookup returning the empty tuple →
  an early return before any context cloning. The empty tuple is a module-level
  singleton, not allocated per call.
- The populated path is O(n) in registered subscribers, iterating a
  pre-sorted frozen tuple (sort happens once at registration, never at
  dispatch).

New dev dependency: `pytest-benchmark`. The perf suite runs in `make check`
and CI and is release-blocking for this subsystem.

## 6. Security

A first-class hookpoint primitive is a significant security surface. The
following are contract decisions this slice locks.

### 6.1 Tier is a requested capability, operator-granted

A plugin's manifest **declares the tier it requests per hookpoint
subscription** — including `system`. There are legitimate reasons to want
system tier: a SIEM/audit forwarder, an org-wide DLP overlay, a compliance
recorder — all must observe or act *before* other hooks and before core
redaction.

At install time the operator is prompted with the **security implication of
the requested tier**, for example:

> Plugin `acme-siem` requests **SYSTEM tier** on `audit.append.before_db_write`.
> At this tier it observes and may mutate every audit row before any other hook
> or core redaction runs, across every persona and user. Approve / Deny.

The operator grants or denies; the grant is recorded and itself audited. The
`CapabilityGate` then **enforces** the granted tier at registration: a plugin
registering at a tier it was not granted is refused and audited (fail-loud).

### 6.2 Two independent gates, both must pass

1. **Publisher-side allow-list** (code policy): the hookpoint's
   `subscribable_tiers` declares which tiers may subscribe at all.
   `secret_broker.get.before_return` (a Slice-3 hookpoint) would be
   `subscribable_tiers={"system"}`, so a `user-plugin` tier can't even request
   it.
2. **Operator-side grant** (install-time trust): the operator approved *this
   specific plugin* for the tier it requested.

A plugin requesting system tier on a system-only hookpoint works only if the
operator explicitly trusted it at install. Defense in depth: a
compromised-but-untrusted plugin cannot silently elevate; a deliberately-
trusted security plugin can do its job.

### 6.3 Re-entry detection

A subscriber on `audit.append.committed` that itself calls `audit.append(...)`
must not infinite-loop. The registry tracks active correlation IDs and the
hookpoints entered under each in a per-task `ContextVar` stack. Re-entry into
the same hookpoint within the same correlation chain dispatches via an
internal `_invoke_internal` path that **skips the hook chain** for the
re-entrant call (and is itself audited as a `hooks.reentry_bypass` system
event). This is the only place the chain is bypassed and it is documented
loudly at the call site.

### 6.4 Per-hook timeout

Each subscriber invocation is wrapped in
`asyncio.wait_for(subscriber(ctx), timeout=HOOK_TIMEOUT)` (default **100ms**,
per-hookpoint configurable). A timeout is **a fault, not a refusal**: it is
logged, audited (`hooks.subscriber_timeout`), and treated as if the subscriber
returned its incoming context unchanged. A hung or runaway subscriber cannot
hang the orchestrator.

### 6.5 Refusal authorization per hookpoint

The publisher declares `refusable_tiers`. A `HookRefusal` raised by a
subscriber whose tier is not in that set is unauthorized: it is converted to a
fail-loud `HookError`, audited, and the action proceeds (the refusal is
rejected, not honoured). For `audit.append.*`, only `system` tier may refuse
(a `user-plugin` refusing to audit would create a silent system failure — the
exact thing CLAUDE.md hard rule #7 forbids).

### 6.6 Frozen context

`HookContext` is frozen; mutations return new instances. Subscribers cannot
mutate shared chain state through the context object. (They can still have
arbitrary Python side effects in their own bodies — irreducible; the
install-time trust decision is the control.)

### 6.7 Audit correlation

Every hook invocation (every subscriber call across every kind) emits an audit
row keyed to the originating action's `correlation_id`. The full hook chain for
any action — including refusals, timeouts, and re-entry bypasses — is
reproducible from the audit log.

### 6.8 Deferred to Slice 3 (when the real gate + MCP transport ship)

- The real `CapabilityGate` implementation (manifest parsing, persisted grants,
  the install-time prompt UX).
- MCP-transport hook registration for out-of-process plugins.
- Data-classification tags per hookpoint (formal T1/T2/T3 labels matched
  against subscriber clearance — depends on the trust-tier work, ADR-0013).
- Audit-volume sampling/aggregation (every hook invocation emitting a full
  audit row could be hundreds/sec once actions carry real subscribers; the
  sampling strategy lands when the volume is measurable).

### 6.9 Out of scope

- Supply-chain / package signing — handled at the `uv.lock` / dependency-pin
  layer, not hook-specific.
- ContextVar mutation guards — speculative without a concrete exploit; Python
  does not enforce import boundaries.

## 7. The proof-of-concept: `audit.append`

`audit.append` is chosen as the single PoC action: it is simple, high-value
(everything audits), already exists in five call sites, and exercises the
full lifecycle-stage shape.

**Changes in `src/alfred/audit/log.py`:**
- New `AuditAppendInput` frozen dataclass mirroring `append`'s kwargs, with
  `with_subject` / `with_result` copy helpers.
- `append` is rewritten to thread three (optionally four) hookpoints:
  - `audit.append.before_validate` — `pre`, all tiers subscribable, all tiers
    refusable (cheap demo of input mutation).
  - `audit.append.before_db_write` — `pre`, `subscribable_tiers={system,
    operator}`, `refusable_tiers={system}` (the security-relevant stage).
  - `audit.append.write_failed` — `error`, observers + optional swallow.
  - `audit.append.committed` — `post`, observers.
- The body otherwise unchanged; with zero subscribers the persisted row is
  byte-identical to today's.

**Integration test:** register a real `system`-tier `pre` subscriber on
`before_db_write` that redacts a subject field; assert the persisted row is
redacted and an audit-correlation row exists for the subscriber invocation.

## 8. Testing strategy

TDD per task. Coverage:

- **Registry unit tests** — register/lookup; order within tier; tier ordering
  (system → operator → user-plugin); duplicate-subscription handling;
  async-only enforcement; `reset()` isolation.
- **Dispatch tests** — `pre` mutation flows to action; `post` mutation;
  `error` swallow-and-replace; `cancel` cleanup-only and cannot suppress.
- **Refusal tests** — `HookRefusal` short-circuits `pre`; action does not run;
  `error` chain observes the refusal; unauthorized refusal (tier not in
  `refusable_tiers`) is rejected + audited.
- **Security tests** (co-owned with `alfred-security-engineer`; a new `hooks`
  category in the adversarial corpus):
  - `user-plugin` tier rejected on a `subscribable_tiers={"system"}` hookpoint.
  - tier not granted by the gate → registration refused + audited.
  - re-entry: a `committed` subscriber calling `audit.append` does not
    infinite-loop and emits a `hooks.reentry_bypass` audit row.
  - per-hook timeout: a 200ms subscriber on a 100ms budget → fault-handled,
    action proceeds, `hooks.subscriber_timeout` audited.
- **Audit-correlation test** — every invocation in a chain shares the action's
  `correlation_id`; the chain is reconstructable from the audit log.
- **PoC integration test** — `audit.append` with three real hookpoints + a
  redacting subscriber; persisted row is redacted.
- **Performance suite** (`pytest-benchmark`) — the three benches in §5.

## 9. ADR-0014 amendment

ADR-0014's prose ("every *action* AlfredOS takes is hookable") can read as an
outer-wrapper-only model. This slice adds a clarifying paragraph to ADR-0014:

> **Core lifecycle methods are themselves hookpoint producers.** They use the
> same `hooks.invoke()` primitive that plugins use to declare extension points.
> There is no asymmetry between "core actions get hooks" and "plugins publish
> hooks" — a hookpoint is a named, string-keyed extension point, and any code
> (core or plugin) may both publish and subscribe. The four kinds (pre / post /
> error / cancel) are routing semantics on an invocation, not the structure-
> defining concept; an action exposes as many or as few hookpoints as its
> author finds useful.

The amendment lands in the spec's documentation PR (PR-C), not as a separate
ADR-0015 — ADR-0014's architectural spirit is unchanged; only the granularity
reading is clarified.

## 13. Open questions for the implementer

- **Registry singleton vs explicit injection.** This spec assumes a per-process
  singleton (module-import registration is the in-tree ergonomics ADR-0014
  calls for). If the implementer finds the singleton fights the no-global-state
  convention in `CLAUDE.md`, an injected registry threaded through bootstrap is
  an acceptable alternative — but the `@hook` decorator must still resolve to
  *some* registry at import time, so the singleton would only become an
  injection seam for tests. Decide at PR-A plan time.
- **`metadata` value type.** `Mapping[str, object]` is proposed (avoids `Any`).
  If subscribers need richer typed metadata, a per-hookpoint metadata type
  parameter could be added — but YAGNI for the PoC.
- **Result carrier for value-returning actions.** `HookContext[T]` carries only
  `input`. The PoC action (`audit.append`) returns `None`, so `post`/`error`
  subscribers have nothing but the input to observe. Actions that return a
  value (a future `provider.call` returning a `CompletionResponse`) will need
  the result threaded into the context for `post`/`error` subscribers to
  observe or mutate it. Two candidate shapes: (a) widen to
  `HookContext[TInput, TOutput]` with an `output: TOutput | None` field
  populated for `post`/`error`; or (b) pass the result as a separate `result=`
  parameter to `invoke` (mirroring how `error` passes `exc=`). Decide when the
  first value-returning action is wired (not in this slice). The contract for
  this slice covers `None`-returning actions only; the PoC does not pre-commit
  the result-carrier shape so the later decision stays unconstrained.
