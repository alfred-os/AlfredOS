# Slice 2.5 — Pluggable Hooks for Every Action: Design

> **Status:** Draft for review (v3 — revised after the PR #112 review team, CodeRabbit re-review, and the /review-plan coordinator gate)
> **Date:** 2026-05-27
> **Slice:** 2.5 (between Slice 2 graduation and Slice 3's MCP-transport rewrite)
> **Implements:** [ADR-0014](../../adr/0014-pluggable-hooks-for-every-action.md) (with a clarifying amendment, see §9)
> **Forward-references:** [ADR-0013](../../adr/0013-defer-t1-t3-and-dual-llm.md) (defer T1/T3 + dual-LLM to Slice 3), [ADR-0009](../../adr/0009-comms-adapter-protocol-slice2-only.md) (CommsAdapter Protocol Slice-2-only → Slice-3 MCP)

## 0. Summary

AlfredOS gains a first-class **hookpoint** primitive: a named, string-keyed
extension point that any code — core lifecycle methods, in-tree plugins, or
(from Slice 3) out-of-process MCP plugins — can both **declare** (publish) and
**subscribe to**. Subscribers run in a deterministic, capability-gated chain
and may mutate the action's input, refuse the action, observe or mutate its
result, react to its errors, or run cleanup on cancellation.

This slice ships the **contract and registry**, instruments **one
proof-of-concept action** (`memory.episodic.record`, with five internal
hookpoints spanning all four kinds — two `pre`, one `error`, one `post`, one
`cancel`) to exercise the full lifecycle shape end-to-end, and lands a
release-blocking **performance gate**. The real capability-gate implementation
and the MCP transport for out-of-process plugins are explicitly **Slice 3**;
this slice ships against a **`DevGate`** (see §6.2) — *not* an allow-everything
gate — because the in-process `@hook` path registers subscribers at import
time *now*, so the gate is load-bearing from day one even before any
out-of-process plugin exists.

**No behaviour change to existing actions** beyond `memory.episodic.record`
gaining internal hookpoints whose chains are empty by default (zero registered
subscribers → a single dict lookup, no observable change; a byte-identical
persisted-row regression test guards this).

> **Why not `audit.append` as the PoC?** An earlier draft used it. The review
> team flagged a recursion hazard: §6.7 mandates every hook invocation emits an
> audit row, so a hook on `audit.append.*` would itself call `audit.append`
> with a *fresh* correlation ID, which the re-entry guard (keyed on the active
> chain) would not catch → unbounded recursion + a quarantine-tripping cascade
> on audit-write failure. `memory.episodic.record` returns `None` (no
> result-carrier complication, see §13), does not call `audit.append` in its
> body (no recursion), and gives us a genuinely security-relevant demo: a
> pre-hook that redacts a leaked secret from episode `content` before it is
> persisted.

## 1. Motivation

ADR-0014 establishes *why* hooks are needed: AlfredOS's household-OS vision
requires third parties (plugins, the operator, system components) to observe or
intercept actions taken by other components without those components knowing
about the third party. The event bus (PRD §5) is observe-only and out-of-band;
the capability gate (PRD §7.1) gates tool *invocation* permission, not
mid-flight interception; the audit log (PRD §7.4) is after-the-fact. None of
these is an in-band interception seam.

This spec turns ADR-0014's architectural commitment into an implementable
contract. The key conceptual move, refined during brainstorming beyond
ADR-0014's original wording: **hookpoints are a general extensibility
primitive, not an outer wrapper around a fixed core-defined action set.** Core
lifecycle methods are themselves hookpoint *producers* — they use the exact
same `invoke()` primitive a plugin uses to declare its own lifecycle. There is
no asymmetry between "core actions get hooks" and "plugins publish hooks." This
is the pytest-pluggy / Django-signals / WordPress-action+filter family of
designs.

PRD §5.1 currently frames the manifest hook surface in outer-wrapper terms.
The "core methods are producers" refinement therefore also needs to land in
the PRD, not just in ADR-0014. Because PRD edits are human-gated (CLAUDE.md
self-improvement rule #4), this refinement ships as a **separate proposed PRD
diff for explicit human approval** (see §9), not silently folded into a code PR.

## 2. Glossary (pointer)

The canonical glossary is `docs/glossary.md`. This slice adds the terms below
to it (PR-C, §10). This section is a **pointer, not a second source of truth** —
definitions live in `docs/glossary.md`; the list here is only the set of terms
this slice introduces:

- **action** — a named unit of work the core or a plugin dispatches (a tool
  call, a provider call, a memory write, a comms outbound, an audit write, a
  persona-to-persona message, a skill invocation). Distinct from a *tool call*
  (one kind of action). Matches ADR-0014's definition.
- **hookpoint** — a named, string-keyed extension point declared at a point in
  some code's execution.
- **hook kind** — one of `pre` / `post` / `error` / `cancel`.
- **hook tier** — one of `system` / `operator` / `user-plugin`.
- **`HookContext[T]`** — the frozen typed payload threaded through a chain.
- **`HookRefusal`** — the exception a `pre` subscriber raises to short-circuit.
- **PoC** — proof-of-concept; here, the single instrumented action
  (`memory.episodic.record`) that exercises the contract end-to-end.

## 3. Architecture

A new top-level subpackage `src/alfred/hooks/`:

```
src/alfred/hooks/
├── __init__.py         # public exports
├── registry.py         # HookRegistry: register subscribers, lookup, ordered chain
├── context.py          # HookContext[T] frozen generic dataclass
├── errors.py           # HookError → {HookRefusal, HookSubscriberError, ...} + t() messages
├── capability.py       # CapabilityGate Protocol + DevGate impl (Slice-2.5 default)
├── audit_sink.py       # AuditSink Protocol + the default structlog-only sink
├── decorators.py       # @hook(name, kind, tier) subscriber-side decorator
└── invoke.py           # invoke() primitive + the `invoking()` publisher helper
```

### 3.1 Public surface (the contract everyone codes against)

```python
from alfred.hooks import (
    hook,            # @hook(name, kind, tier) — subscriber decorator
    invoke,          # await invoke(name, ctx, kind=...) — the dispatch primitive
    invoking,        # async with invoking(action_id, input) as flow — publisher helper
    HookContext,     # HookContext[T] frozen payload
    HookError,       # base of the hook exception taxonomy
    HookRefusal,     # raised by a pre-hook to short-circuit
    HookSubscriberError,  # wraps an unexpected subscriber exception (see §6.6)
    CapabilityGate,  # Protocol the registry consults at registration
    DevGate,         # Slice-2.5 default impl (refuses `system` unless opted in)
    AuditSink,       # Protocol the dispatch engine writes hook-trace rows through
    HookRegistry,    # the registry type (injected; see §3.2)
    get_registry,    # process-default registry accessor
    set_registry,    # bootstrap swap + test seam (hot-reload itself is Slice 3, §3.2)
)
```

### 3.2 The registry — injected, not a hard singleton

`HookRegistry` is an instance, not a module global. A process-default instance
is created lazily and accessed via `get_registry()`; `set_registry(new)` swaps
it. This addresses the no-global-state convention now and reserves the seam a
future hot-reload needs (without claiming to implement hot-reload this slice):

- **No-global-state convention (CLAUDE.md).** Production wiring threads the
  registry through bootstrap; `get_registry()` is the fallback for the
  module-import-time `@hook` decorator, which must resolve to *some* registry
  at import.
- **Bootstrap + test seam (Slice 2.5 scope).** In 2.5, `set_registry()` is a
  **bootstrap-and-test seam only** — production wiring constructs the registry
  once at bootstrap and installs it; tests construct fresh instances per test.
  It is *not* an in-process hot-reload mechanism this slice. **In-process
  hot-reload is explicitly deferred to Slice 3** for a mechanism reason the
  review surfaced: a plain re-import of an already-imported module is a
  `sys.modules` no-op (the `@hook` decorators do not re-run, so a naive
  rebuild-and-swap would install an *empty* registry — silent subscriber loss),
  and `importlib.reload()` re-fires every module-level side effect, not just the
  decorators. PRD §5's supervisor governs crashed-**subprocess** restart (with
  backoff/circuit-breaker), not in-process module reload — so there is no
  existing core mechanism to lean on for live reload. Slice 3, which moves
  plugins to the out-of-process MCP transport, is where reload becomes a
  subprocess-restart concern and gets a real design. The registry still records
  each subscriber's `origin_module` (for audit + deterministic rebuild when that
  Slice-3 design lands); the clean property that holds *today* is that dropping a
  registry instance drops its subscribers with no stale references or
  double-registration (they are instance-held, not global).

Tests use a `fresh_registry()` fixture that constructs a new instance and
`set_registry()`-installs it for the test's duration, then restores the prior
one — no cross-test leakage.

Responsibilities:
- `register(subscriber)` — validate (async-only, gate-approved tier, hookpoint
  accepts the requested tier), store keyed by `(hookpoint, kind)`, sorted by
  `(tier_rank, registration_seq)`.
- `subscribers_for(hookpoint, kind) -> tuple[Subscriber, ...]` — O(1) dict
  lookup returning the ordered, frozen tuple. The empty case returns a
  module-level `_EMPTY: tuple[Subscriber, ...] = ()` singleton (asserted by
  object identity in a test so the no-subscriber path provably allocates
  nothing).
- `reset()` / construction — test isolation.

The registry holds **no per-invocation state**. Invocation state (the
correlation-ID re-entry stack) lives in a `ContextVar` so it is per-task,
matching AlfredOS's existing per-coroutine `_active_lang` pattern
(`src/alfred/i18n/translator.py`).

### 3.3 HookContext

```python
@dataclass(frozen=True, slots=True)
class HookContext[T]:
    action_id: str               # logical action, e.g. "memory.episodic.record"
    hookpoint: str               # specific point, e.g. "memory.episodic.record.before_db_write"
    input: T                     # the typed action input
    correlation_id: str          # UUID shared across the whole chain
    kind: Literal["pre", "post", "error", "cancel"]
    metadata: Mapping[str, object] = field(default_factory=dict)

    def with_input(self, new_input: T) -> Self:
        return replace(self, input=new_input)

    def with_metadata(self, **kv: object) -> Self:
        return replace(self, metadata={**self.metadata, **kv})

    def for_stage(self, *, hookpoint: str, kind: HookKind) -> Self:
        """Return a NEW context retargeted to a stage. Frozen — never mutates."""
        return replace(self, hookpoint=hookpoint, kind=kind)
```

Frozen + copy methods: subscribers cannot mutate shared state through the
context. `invoke` is authoritative for `kind`/`hookpoint` and applies
`for_stage()` to produce a **new** instance reflecting the current stage before
dispatch (it does not — cannot — rewrite the frozen fields in place). `T` is a
per-action input dataclass; convention: every hookpoint-publishing action ships
an `<ActionName>Input` frozen dataclass. For the PoC, `EpisodicRecordInput`
lives in `src/alfred/memory/episodic.py`.

### 3.4 Publisher side — `invoke` primitive + `invoking` helper

The low-level primitive:

```python
type HookKind = Literal["pre", "post", "error", "cancel"]

async def invoke[T](
    name: str,
    ctx: HookContext[T],
    *,
    kind: HookKind,
    subscribable_tiers: frozenset[str] = frozenset({"system", "operator", "user-plugin"}),
    refusable_tiers: frozenset[str] = frozenset({"system", "operator", "user-plugin"}),
    fail_closed: bool = False,
    exc: BaseException | None = None,
) -> HookContext[T]:
    ...
```

- `subscribable_tiers` — which tiers MAY have subscribers on this hookpoint
  (registration-time enforced; re-checked at invoke).
- `refusable_tiers` — which tiers MAY raise `HookRefusal` here (see §6.5).
- `fail_closed` — when `True`, a subscriber timeout or unexpected exception on
  this hookpoint **refuses the action** rather than passing through (see §6.4,
  §6.6). Security-relevant pre-hookpoints set this.
- `exc` — typed `BaseException | None` (not `Exception` — `CancelledError` is a
  `BaseException`); required when `kind="error"` or `kind="cancel"`.

Because hand-threading four `invoke` calls + a `uuid4()` per action is
error-prone (a forgotten `exc=`, a mis-threaded `ctx`, a dropped stage on early
return), the **`invoking` helper** is the ergonomic surface most action authors
use:

```python
async def record(self, *, user_id: str, role: Role, content: str, ...) -> None:
    inp = EpisodicRecordInput(user_id=user_id, role=role, content=content, ...)
    async with invoking("memory.episodic.record", inp) as flow:
        await flow.pre("before_validate")
        _validate(flow.input)
        await flow.pre(
            "before_db_write",
            subscribable_tiers={"system", "operator"},
            refusable_tiers={"system"},
            fail_closed=True,          # a timed-out/erroring redactor must NOT write
        )
        async with flow.body():        # body() emits write_failed on exception, after_flush on success
            await self._persist(flow.input)
```

`invoking` mints the correlation_id once, threads the (frozen) context between
stages, auto-emits the `error` chain (`*.write_failed`-style) on a body
exception and the `post` chain (`*.after_flush`-style) on success, and fires the
`cancel` chain on `CancelledError`. `invoke` remains public for callers that
need finer control. Dispatch internally routes to four private handlers
(`_run_pre` / `_run_post` / `_run_error` / `_run_cancel`) sharing a
`_run_chain` core, so each kind's branch coverage is independently assertable
(required by the 100% gate, §8).

### 3.5 Subscriber side — `@hook`

```python
type HookFn[T] = Callable[..., Awaitable[HookContext[T] | None]]

@hook("memory.episodic.record.before_db_write", kind="pre", tier="system")
async def redact_secrets(ctx: HookContext[EpisodicRecordInput]) -> HookContext[EpisodicRecordInput]:
    return ctx.with_input(replace(ctx.input, content=_scrub(ctx.input.content)))

@hook("memory.episodic.record.write_failed", kind="error", tier="system")
async def alert_on_write_failure(
    ctx: HookContext[EpisodicRecordInput], exc: BaseException
) -> HookContext[EpisodicRecordInput] | None:
    _alert(ctx.correlation_id, exc)
    return None   # None → re-raise (the default); return a context → swallow-and-substitute
```

- All subscribers are `async def`. A sync function passed to `@hook` is rejected
  **at decoration time** with `HookError` carrying the `t()` message
  `t("hooks.subscriber_must_be_async", name=fn.__qualname__)` →
  *"Hook subscriber {name} must be `async def`. Add `async` to the function
  definition."* (operator/developer-facing, so it goes through `t()` per i18n
  rule #1).
- `pre` subscribers return a `HookContext` (mutated or pass-through) or raise
  `HookRefusal`.
- `post` subscribers return a `HookContext` (may annotate `metadata` or mutate
  `input` for downstream observers) or pass through (`return None` keeps the
  incoming context).
- `error` subscribers take `(ctx, exc: BaseException)` and return a
  `HookContext` to swallow-and-substitute, or `None` to re-raise (the default).
- `cancel` subscribers take `(ctx, exc: BaseException)`, run cleanup only,
  return `None`; they cannot suppress the `CancelledError`.

Return-type contract: every kind's subscriber returns `HookContext[T] | None`.
`None` always means "no change / proceed with default disposition"; a returned
context means "use this." This single rule replaces the v1 draft's
contradictory `-> None` example.

### 3.6 Plugins as hookpoint publishers + the manifest surface

A plugin exposes its own lifecycle by calling `invoke()`/`invoking()` from its
own code, exactly as core does. Subscriptions + published hookpoints are
declared in the plugin manifest using the **existing `hooks` block** that PRD
§5.1 and ADR-0014 already specify — a list of `(action, kind)` tuples the
plugin handles, plus the tier it requests for each:

```toml
[[hooks]]
action = "memory.episodic.record.before_db_write"
kind = "pre"
tier = "operator"
```

This slice does **not** invent new manifest field names (an earlier draft did;
the review flagged the drift against PRD §5.1 / ADR-0014). The manifest is
consumed by the Slice-3 MCP transport; this slice only documents the shape so
Slice 3 implements against one definition.

## 4. The four kinds and the dispatch sequence

A publishing action threads its lifecycle through named hookpoints via
`invoking` (§3.4). Conceptually, with the helper desugared:

```python
async def record(self, *, user_id, role, content, ...) -> None:
    ctx = HookContext(
        action_id="memory.episodic.record",
        hookpoint="memory.episodic.record.before_validate",
        input=EpisodicRecordInput(user_id=user_id, role=role, content=content, ...),
        correlation_id=str(uuid4()),
        kind="pre",
    )
    ctx = await invoke("memory.episodic.record.before_validate", ctx, kind="pre")
    _validate(ctx.input)
    ctx = await invoke(
        "memory.episodic.record.before_db_write", ctx, kind="pre",
        subscribable_tiers={"system", "operator"},
        refusable_tiers={"system"},
        fail_closed=True,
    )
    try:
        await self._persist(ctx.input)
    except asyncio.CancelledError as exc:
        # Cancellation is handled FIRST and separately: it runs the cancel
        # chain (cleanup-only) and is ALWAYS re-raised — never routed to the
        # error chain, never swallowed (§6.4). Catching it before the generic
        # handler is load-bearing; CancelledError is a BaseException, so a bare
        # `except BaseException` would wrongly send it to `write_failed`.
        await invoke("memory.episodic.record.cancelled", ctx, kind="cancel", exc=exc)
        raise
    except Exception as exc:
        await invoke("memory.episodic.record.write_failed", ctx, kind="error", exc=exc)
        raise
    else:
        await invoke("memory.episodic.record.after_flush", ctx, kind="post")
```

(The `invoking` helper performs exactly this split internally — `cancel`
before `error` — so action authors using the helper cannot wire it the wrong
way; the desugaring above is shown only to make the contract explicit.)

**`pre` chain:** subscribers run in tier order, each receiving the prior's
outgoing context. The first `HookRefusal` short-circuits: the action does not
run, no further `pre` subscribers run, the refusal is audited, and `HookRefusal`
propagates to the caller. A mutation by an earlier subscriber is **discarded**
when a later subscriber refuses (the action never sees it — tested explicitly).

**`post` chain:** runs after the action succeeds; mutations are for downstream
chain observers.

**`error` chain:** runs when the action raises. A subscriber returns a
replacement context to swallow-and-substitute, else re-raises. Suppression of a
real failure is itself audited loudly (`hooks.error_suppressed`), the original
exception is chained (`raise ... from exc`), and suppression is **system-tier-
only** on persistence hookpoints so a user-plugin can't mask a DB-write failure
(per CLAUDE.md hard rule #7).

**`cancel` chain:** runs on `asyncio.CancelledError`. Cleanup only; cannot
suppress. The publisher distinguishes an *external* cancellation (propagate +
fire cancel chain) from a chain-deadline *timeout* (§6.4).

## 5. Performance

Per ADR-0014, hook dispatch is on hot paths and must be near-free with no
subscribers.

**What is measured.** The budget bounds **dispatch overhead only** — the
`invoke()` machinery — *excluding* the action body and its DB round-trip (which
dominate real latency; the gate is an overhead gate, not a throughput gate).
Each bench measures **delta over a same-loop baseline** (`await` of a no-op
coroutine) so the gate is not CI-hardware-absolute and not dominated by
event-loop scheduling noise. `pytest-benchmark` is pinned (`warmup`, fixed
`rounds`/`iterations`, p99 metric).

**Budget (release-blocking):**
- Empty hookpoint (zero subscribers), measured at a **real publisher call site**
  (the full `record()` body minus the DB write): **p99 < 10µs** delta over
  baseline. A separate micro-bench reports `HookContext` construction +
  `uuid4()` cost on its own (informational, not gated — it is intrinsic to the
  publisher, not the dispatch engine).
- 5-subscriber `pre` chain: **p99 < 100µs** delta over baseline.
- Refusal short-circuit: subscribers after the refusing one do not run
  (correctness assertion paired with the bench).

**Timeout mechanism vs the budget.** The timeout is a **per-chain deadline**,
not a per-subscriber budget. Per-subscriber `asyncio.wait_for` carries real
overhead (Task + timer handle, ~tens of µs each) that would blow the 100µs
budget at 5 subscribers, so the dispatch engine uses **one** `asyncio.timeout()`
wrapping the whole chain at a hookpoint. There is deliberately no per-subscriber
timer (see §6.4 for the exact contract). The chain-deadline default is **250ms**
per hookpoint (configurable per hookpoint); this perf budget (the 10µs/100µs
dispatch-overhead gate) is a separate measurement from that wall-clock deadline.

**No-alloc guarantees:** the no-subscriber path is a single dict lookup
returning the `_EMPTY` tuple singleton, with an early return before the re-entry
ContextVar is even read (that read is skipped when the chain is empty).

New dev dependency: `pytest-benchmark`, release-blocking for this subsystem.

## 6. Security

A first-class hookpoint primitive is a significant security surface. The
following are contract decisions this slice locks.

### 6.1 Tier is a requested capability, operator-granted

A plugin's manifest **declares the tier it requests per hookpoint
subscription** — including `system`. There are legitimate reasons to want
system tier: a SIEM/audit forwarder, an org-wide DLP overlay, a compliance
recorder.

At install time the operator is prompted with the **security implication of the
requested tier**. The prompt text is operator-facing UI and ships as `t()` keys
with `plugin` / `tier` / `hookpoint` as kwargs (rule #1), even though the prompt
UX itself lands in Slice 3 (§6.8). Example rendering:

> Plugin `acme-siem` requests **SYSTEM tier** on
> `memory.episodic.record.before_db_write`. At this tier it observes and may
> mutate every recorded turn before any other hook or core redaction runs,
> across every persona and user. Approve / Deny.

The grant is recorded and itself audited. The `CapabilityGate` enforces the
granted tier at registration: a plugin registering at a tier it was not granted
is refused and audited (fail-loud).

### 6.2 The Slice-2.5 gate is `DevGate`, not allow-everything

The in-process `@hook` path registers subscribers **at import time, now** —
before any out-of-process plugin exists. So the gate is load-bearing this slice,
not deferred. Slice 2.5 ships **`DevGate`**:

- grants `operator` and `user-plugin` tiers freely (in-tree code is trusted-by-
  being-in-tree — effectively T0; stated explicitly as the slice's trust
  assumption);
- **refuses `system` tier** unless explicitly opted in via the
  **constructor only** — `DevGate(allow_system=True)`.

**No ambient env flag.** An earlier draft also allowed `ALFRED_HOOKS_ALLOW_SYSTEM`;
the review removed it (sec-007). A process-wide env switch is an ambient
escalation hazard — a leaked CI export, a compose default, or a dependency
reading `os.environ` at import could silently grant the highest-blast tier to
every in-tree `@hook` decorator before any reviewer gate, and `DevGate` treats
in-tree as T0. Constructor-only keeps the grant an explicit, code-local,
single-call-site decision the bootstrap wiring owns. If a future operator
genuinely needs an env-driven grant, it is read **once at bootstrap** into the
injected gate (never per-registration), emits a loud audit row when it enables
system, stays out of `.env.example`/compose, and tests set it via constructor +
monkeypatch-with-teardown only (never a global export — cf. the PR #108
`_pg_stack` env-clobber regression). That is a Slice-3 concern, not 2.5.

This gives PR-A's deny-path security tests a real refusal to assert against
(an always-allow gate would make those tests vacuous — rule #4), and prevents
an in-tree module (or a compromised dependency) from silently grabbing the
highest-blast tier. The PoC's system-tier redactor runs in tests with
`DevGate(allow_system=True)` constructed explicitly, exercising the granted
path. Slice 3 replaces `DevGate` with the real manifest-driven gate + install
prompt.

### 6.3 Two independent gates, both must pass

1. **Publisher-side allow-list** (code policy): the hookpoint's
   `subscribable_tiers`.
2. **Operator-side grant** (the `CapabilityGate`): the operator approved *this
   plugin* for the tier it requested.

### 6.4 Chain timeout — fail-closed where it matters

The timeout contract is a **per-hookpoint chain deadline**, NOT a per-subscriber
budget. The whole chain at a hookpoint must complete within the deadline
(default **250ms**, configurable per hookpoint), enforced by a single
`asyncio.timeout()` wrapping the chain (§5 — there is no per-subscriber timer,
for the perf reason given there). When the deadline expires, the in-flight
subscriber is cancelled (its coroutine awaited to completion so it cannot leak a
half-open cursor/lock) and the hookpoint's `fail_closed` policy applies:

- On a **`fail_closed=False`** hookpoint (observe-only `post` hooks, advisory
  `pre` hooks): a deadline expiry is a **fault, not a refusal** — logged +
  audited (`hooks.chain_timeout`) + the chain is abandoned and the action
  proceeds with the last good context.
- On a **`fail_closed=True`** hookpoint (security-relevant `pre` hooks like the
  PoC redactor): a deadline expiry **refuses the action** (`HookError`,
  audited). A timed-out redactor must never let the un-redacted write proceed.

A consequence of the chain-level deadline: a single slow subscriber can consume
the whole chain's budget, starving later subscribers. That is the accepted
trade-off for avoiding per-subscriber timer overhead (§5); operators who need
per-subscriber isolation are pointed to the Slice-3 out-of-process transport
where subscribers run in their own process.

The dispatch engine awaits the cancelled subscriber coroutine to completion
before continuing the chain, so a timed-out subscriber cannot leak a half-open
cursor/lock into the next subscriber. An *external* `CancelledError` (the
publisher task itself being cancelled) is never swallowed: it fires the cancel
chain and propagates.

### 6.5 Refusal authorization per hookpoint

The publisher declares `refusable_tiers`. A `HookRefusal` from a tier not in
that set is **unauthorized**: it is logged + audited (`hooks.unauthorized_refusal`),
the refusal is **ignored** (not honoured — the action proceeds normally), and
**no exception propagates to the caller**. "Fail-loud" here means the audit row
+ log, not a raised exception — the action behaves as if the unauthorized hook
had passed through. (This resolves the v1 ambiguity the review flagged: the
caller is not handed a `HookError` for a hook it didn't write.)

### 6.6 Unexpected subscriber exceptions

A subscriber may raise something that is neither `HookRefusal` nor a timeout — a
plain `ValueError` from a buggy hook. This is **never silently swallowed**
(rule #13). The dispatch engine:

- wraps it as `HookSubscriberError` (chaining the original via `from exc`);
- emits a loud `hooks.subscriber_error` audit row keyed to the correlation ID;
- then applies the hookpoint's `fail_closed` policy: `fail_closed=True` →
  the action is refused (the wrapped error propagates); `fail_closed=False` →
  the subscriber is treated as pass-through and the chain continues (the error
  is still audited loudly — it is recorded, not hidden).

This closes the v1 gap (the `error` chain only covered the *action* raising, not
a *subscriber* raising) that the Critical finding identified.

### 6.7 Hook-performed outbound stays DLP-gated

A system-tier hook that observes pre-redaction content (the SIEM example) and
then sends it anywhere external is, by construction, an **outbound egress** and
remains subject to the DLP rule (CLAUDE.md rule #6 / PRD §7.1): such a hook is a
**declared DLP-egress sink**, its manifest declares the egress, and an
adversarial test asserts its outbound passes through the redactor. Pre-redaction
*observation* inside the process is permitted; pre-redaction *exfiltration* is
not a hook carve-out from DLP.

### 6.8 Audit correlation + volume

Every hook invocation emits an audit row keyed to the originating action's
`correlation_id` (refusals, timeouts, unauthorized refusals, subscriber errors,
re-entry bypasses all included), via the `AuditSink` (§3.1). The full chain for
any action is reproducible from the audit log.

The PoC's `AuditSink` writes through the real `audit.append` path. Because the
PoC action is `memory.episodic.record` (not `audit.append`), there is **no
recursion**: a hook-trace audit row is an ordinary `audit.append` call that does
not re-enter `memory.episodic.record`. (Had the PoC been `audit.append`, the
sink would need a chain-skipping `_append_raw` to avoid recursion; choosing a
non-audit PoC removes the hazard entirely.)

### 6.9 Re-entry detection

A subscriber that re-enters the *same* hookpoint within the *same* correlation
chain (e.g. an `after_flush` hook on action X that calls action X again) dispatches
via an internal `_invoke_internal` path that **skips the chain** and emits a
`hooks.reentry_bypass` audit row. This is the only place the chain is bypassed;
it is documented loudly and tested. The bypass skips **all** tiers including
system — so a system-tier security hook does not run on the re-entrant call;
this is a deliberate T0-only invariant (the re-entrant call is core-internal),
stated explicitly and covered by a test (including the `asyncio.create_task`
ContextVar-propagation edge: a subscriber that spawns a task does not inherit
the re-entry stack, so it is treated as a fresh chain — documented).

### 6.10 Deferred to Slice 3

- The real manifest-driven `CapabilityGate` + install-time prompt UX.
- MCP-transport hook registration for out-of-process plugins.
- Data-classification tags per hookpoint (T1/T2/T3 labels matched against
  subscriber clearance — depends on the trust-tier work, ADR-0013). **Slice 3
  must treat this as a blocking gate on enabling out-of-process subscribers,
  not a parallel deferral** — an MCP plugin subscribing to a hookpoint that
  carries T3 content without a classification check would be a trust-boundary
  hole.
- Audit-volume sampling/aggregation (hook-trace rows could reach hundreds/sec
  once actions carry real subscribers; sampling lands when volume is
  measurable).

### 6.11 Out of scope

- Supply-chain / package signing — handled at the `uv.lock` layer.
- ContextVar mutation guards — speculative without a concrete exploit.

## 7. The proof-of-concept: `memory.episodic.record`

`EpisodicMemory.record` (`src/alfred/memory/episodic.py`) is the PoC: it returns
`None` (no result-carrier complication), does not call `audit.append` (no
recursion), and persists user `content` + `trust_tier` + `language` to Postgres
— so a pre-hook redacting a leaked secret from `content` before the write is a
genuinely security-relevant demo.

**Changes:**
- New `EpisodicRecordInput` frozen dataclass mirroring `record`'s kwargs —
  including **`language`** (BCP-47, i18n rule #3) and `trust_tier`, so the
  rewritten `record` cannot silently drop the `language` column. Copy helpers
  via `replace()`.
- `record` rewritten to thread five hookpoints via `invoking` (one per stage,
  spanning all four kinds):
  - `memory.episodic.record.before_validate` — `pre`, all tiers,
    `fail_closed=False`.
  - `memory.episodic.record.before_db_write` — `pre`,
    `subscribable_tiers={system, operator}`, `refusable_tiers={system}`,
    **`fail_closed=True`** (the security stage).
  - `memory.episodic.record.write_failed` — `error`.
  - `memory.episodic.record.after_flush` — `post`. **Named `after_flush`, not
    `committed`, on purpose** (mem-1): `record` does `session.add()` +
    `session.flush()` and returns; the durable **commit** happens later in the
    *caller's* transaction (`db.py` `session_scope` wraps the whole orchestrator
    turn). Firing a hook called `committed` right after the flush would let a
    compliance/SIEM subscriber assert durability for a row a later same-turn
    failure can still roll back. `after_flush` is the truthful name; a real
    after-commit signal is a separate, later concern.
  - `memory.episodic.record.cancelled` — `cancel`, fired by `invoking` if the
    action is cancelled mid-flight (cleanup-only; never suppresses). The PoC
    registers no cancel subscriber — cancel-chain *dispatch* is covered by
    PR-A's unit tests — but wiring it proves the helper handles all four kinds.
- Body otherwise unchanged; with zero subscribers the persisted `Episode` row is
  byte-identical to today's (a golden-row regression test asserts this,
  including that `language` survives).

**Integration test:** a real system-tier `pre` subscriber on `before_db_write`
(registered against a `DevGate(allow_system=True)`) redacts a secret-shaped
substring from `content`; assert the persisted row's `content` is redacted, the
`language` column is intact, and a hook-trace audit row shares the action's
`correlation_id`.

## 8. Testing strategy

TDD per task. **Coverage gate:** the hooks subsystem is a trust boundary (it is
the in-band capability gate for subscriber registration and wraps a persistence
action). PR-A ships `--cov-branch --cov-fail-under=100` on
`src/alfred/hooks/{registry,invoke,capability,decorators,context,errors}.py`,
mirroring the secret-broker (PR-C) precedent.

Coverage:

- **Registry** — register/lookup; tier-then-registration order; duplicate
  handling; async-only enforcement (+ the `t()` message); `fresh_registry()`
  isolation; `set_registry()` swap; `origin_module` tracking; `_EMPTY`
  object-identity (no-alloc proof).
- **Dispatch** — `pre` mutation flows to action; `pre` mutation **discarded on a
  later refusal**; `post` mutation; `error` swallow-and-substitute **and**
  re-raise (`None`); `cancel` cleanup-only for both normal-return and raising
  cancel subscribers (cannot suppress); the four private handlers each branch-
  covered.
- **Fault semantics** — unexpected subscriber exception → `HookSubscriberError`
  + `hooks.subscriber_error` audit + `fail_closed` policy applied (refuse vs
  pass-through); timeout fault vs fail-closed refusal; unauthorized refusal
  ignored + audited + no exception to caller; error-suppression audited +
  chained + system-tier-only on persistence hookpoints.
- **Security** (co-owned with `alfred-security-engineer`; new `hooks` adversarial
  category):
  - `user-plugin` tier rejected on `subscribable_tiers={"system"}`.
  - `DevGate` refuses `system` without the opt-in → registration refused +
    audited; with opt-in → granted path runs.
  - re-entry: an `after_flush` subscriber calling the same action does not loop;
    `hooks.reentry_bypass` row emitted; system tier skipped on the re-entrant
    call.
  - fail-closed timeout: a slow redactor on `before_db_write` refuses the write
    (no un-redacted row persisted).
  - DLP-egress: a hook that performs outbound has its egress pass the redactor.
- **PoC integration** — §7.
- **Performance** (`pytest-benchmark`) — §5; structural assertions (refusal
  short-circuit, no-alloc identity) are separated from the absolute-µs timing
  gate, and the timing gate is delta-over-baseline so it is not CI-flaky.
- **i18n** — `HookError`/`HookRefusal` operator-facing messages resolve through
  `t()`; the sync-function rejection message is a catalog key; the PoC carries
  `language` and a test asserts it survives a `before_db_write` mutation.

**Adversarial corpus `hooks` category (PR-B).** The existing
`tests/adversarial/payload_schema.py` is a closed set of `Literal`s. Adding the
category requires: a `hooks` member to the `Category` literal; an `hk` prefix in
`_PREFIX_TO_CATEGORY` + `_ID_PATTERN`; a row in
`.rulesync/skills/alfred-adversarial-corpus/SKILL.md`; and at least one payload
(`tests/adversarial/hooks/tier_escalation.yaml`) modelling a user-plugin
attempting to register at `system` tier / refuse a system-only hookpoint, with
`expected_outcome: refused`. PR-B enumerates these schema edits explicitly (it
is a coordinated change with the schema's owner).

## 9. ADR-0014 amendment + PRD §5.1 proposed edit

Two doc changes, handled differently because of governance:

1. **ADR-0014 amendment (in PR-C, code-PR-gated).** ADR-0014 is currently status
   *Proposed*; this slice's shipping is the trigger to move it to *Accepted* and
   add the clarifying paragraph:

   > **Core lifecycle methods are themselves hookpoint producers.** They use the
   > same `invoke()`/`invoking()` primitive that plugins use to declare extension
   > points. There is no asymmetry between "core actions get hooks" and "plugins
   > publish hooks" — a hookpoint is a named, string-keyed extension point, and
   > any code (core or plugin) may both publish and subscribe. The four kinds
   > (pre/post/error/cancel) are routing semantics on an invocation, not the
   > structure-defining concept.

2. **PRD §5.1 edit (separate proposed diff, human-gated).** PRD §5.1 currently
   frames the hook surface in outer-wrapper terms. The same "core methods are
   producers" refinement must land there for the PRD to stay source-of-truth.
   Per CLAUDE.md self-improvement rule #4 (PRD edits are human-gated), this is
   prepared as a **standalone proposed diff for explicit human approval**, not
   folded silently into a code PR. The manifest `hooks` block wording in PRD
   §5.1 is **not** changed (this spec conforms to it).

## 13. Open questions for the implementer

- **`invoking` helper API surface.** The desugaring in §3.4 is illustrative;
  PR-A's plan pins the exact `flow.pre()` / `flow.body()` signatures. The
  primitive (`invoke`) is the locked contract; the helper is ergonomic sugar
  over it and may be refined at PR-A plan time without changing the primitive.
- **`metadata` value type.** `Mapping[str, object]` (avoids `Any`). A
  per-hookpoint typed metadata parameter is YAGNI for the PoC.
- **Result carrier for value-returning actions — shape pre-reserved.** The PoC
  returns `None`. To avoid a future breaking change, the public surface reserves
  the name now: a later `HookContext` gains an `output: TOutput | None` field
  (or `invoke` gains a `result=` parameter mirroring `exc=`). PR-A documents the
  reservation so the `post`/`error` carrier can be added in the slice that wires
  the first value-returning action without churning the `invoke` signature. No
  implementation this slice.
