# `security.quarantined.extract` hookpoint — design

**Date:** 2026-06-04
**Author:** Claude Code (on behalf of Ian Dominey)
**Scope:** Issue [#158](https://github.com/alfred-os/AlfredOS/issues/158) — Slice-3 UAT finding E13
**Anchors:** Slice-3 design spec §6.5 (lines 461–476); §7.1 line 82 (DLP placement); §14 line 1159 (hookpoint table); ADR-0017 (slice-3 trust-tier completion)
**Decision (operator):** Option A — wire the hookpoint per spec §6.5. Pattern-matchable exfil channel closed via `OutboundDlp.scan(model_dump)`; semantic-exfil residual risk documented inline.

---

## 1. What this is for

`QuarantinedExtractor.extract` deserialises a structured `ExtractionResult` from the quarantined-LLM plugin. The result is shape-validated via Pydantic. But a free-text `str` field in the schema (e.g. `summary: str`, `error_message: str`) carries content generated by a model that saw T3 content — **that text is an exfiltration channel**.

The spec (§6.5 line 476) names the defence: a `post`-kind subscriber on a `security.quarantined.extract` hookpoint runs `OutboundDlp.scan(result.model_dump())` and refuses on canary trip. Today the hookpoint is **not registered** and no DLP scan runs.

This PR registers the hookpoint and wires the DLP-on-extract subscriber. Closes the pattern-matchable portion of the exfil channel (API keys, canary tokens, credit cards). The **semantic** exfil sub-channel (paraphrased PII in natural language) stays open and is **explicitly documented** as a known limitation of regex-based DLP — out of scope here.

## 2. Architecture

### 2.1 Hookpoint registration

Add at the bottom of `src/alfred/security/quarantine.py` (matches the precedent at `alfred.memory.episodic`, `alfred.identity._ingest`, `alfred.security.capability_gate.proposals`):

```python
def declare_hookpoints(registry: HookRegistry | None = None) -> None:
    """Register the `security.quarantined.extract` hookpoint (spec §6.5).

    Called at module-init time so a CLI invocation or test fixture that
    imports `alfred.security.quarantine` finds the hookpoint pre-declared.
    Idempotent — re-import does not double-register.
    """
    target = registry if registry is not None else get_registry()
    target.register_hookpoint(
        name="security.quarantined.extract",
        subscribable_tiers=SYSTEM_OPERATOR_TIERS,
        refusable_tiers=SYSTEM_ONLY_TIERS,
        fail_closed=True,
    )


declare_hookpoints()  # at module bottom — runs at import time
```

Manifest update: add `"security.quarantined.extract"` to `alfred.security.quarantine` entry in `src/alfred/hooks/_known_hookpoints.py` (extends the #151 work).

### 2.2 Dispatch through the hookpoint chain

Wrap the existing `extract` body in `invoking()` per the `alfred.memory.episodic.record` precedent (`src/alfred/memory/episodic.py:259`):

```python
async with invoking("security.quarantined.extract", inp) as flow:
    flow = await flow.pre(
        "security.quarantined.extract",
        subscribable_tiers=SYSTEM_OPERATOR_TIERS,
        refusable_tiers=SYSTEM_ONLY_TIERS,
        fail_closed=True,
    )

    # ... existing extract body: schema validation, dispatch, audit row emit ...
    result = ExtractionResult(...)

    # post: DLP subscriber runs OutboundDlp.scan(result.model_dump())
    flow = await flow.post(
        "security.quarantined.extract",
        subscribable_tiers=SYSTEM_OPERATOR_TIERS,
        refusable_tiers=SYSTEM_ONLY_TIERS,
        fail_closed=True,
        # Pass the model_dump as part of the flow context so subscribers
        # see the validated payload.
        with_input=result.model_dump(),
    )

    return result
```

`error` kind dispatches via the `invoking()` context manager's exception path automatically (the helper catches and re-raises through the error chain).

**Implementation note** (CR-158 NIT #12): the snippet above uses the `async with invoking(...): flow.pre(); flow.post()` shape because that's the precedent at `alfred.memory.episodic.record`. The `security.quarantined.extract` implementation uses direct `await invoke(...)` calls and a private `_dispatch_error_chain` helper rather than `Flow.body` because the pre and post stages carry **different input payloads** (the post stage needs `result.model_dump()`; the pre stage carries closed-vocab attribution only — `schema_name` + `handle_id`, never the wire payload). The `Flow` helper assumes one input across stages; the two-payload requirement makes it the wrong abstraction for this hookpoint. The end-to-end contract is identical (pre-then-body-then-post; exceptions dispatch the error chain); only the call shape differs.

### 2.3 `OutboundDlpExtractSubscriber`

New module `src/alfred/security/_extract_dlp_subscriber.py`:

```python
"""System-tier DLP subscriber on the `security.quarantined.extract` post chain.

Spec §6.5 line 476 mandates this scan: a typed-extraction schema with a
free-text str field is an exfiltration channel; running OutboundDlp.scan
on the result.model_dump() catches pattern-matchable secrets (canary
tokens, API keys, credit cards) before the validated payload returns to
the orchestrator.

**Known limitation (semantic-exfil residual risk):** OutboundDlp is regex-
based. Arbitrary semantic exfiltration ("the user lives at 123 Main St"
embedded in a natural-language summary field) is NOT caught here.
Closing that channel requires an AI-based DLP layer which is out of
AlfredOS's current threat model. See issue #158 for the discussion.
"""

from __future__ import annotations

import json
from typing import Final

from alfred.security.dlp import OutboundDlp


class OutboundDlpExtractSubscriber:
    """`post`-kind subscriber on `security.quarantined.extract`."""

    _SCAN_ID: Final[str] = "security.quarantined.extract.post.dlp"

    def __init__(self, *, outbound_dlp: OutboundDlp) -> None:
        self._dlp = outbound_dlp

    async def __call__(self, ctx: HookContext) -> HookContext:
        """Scan `ctx.input` (the result.model_dump dict) for DLP triggers.

        On clean scan: return ctx unchanged.
        On canary/DLP trip: raise `HookRefusal` so the dispatch chain
        aborts the extract — the orchestrator never sees the validated
        model_dump.
        """
        # Serialise the dict to JSON for the regex scan. JSON shape is
        # the canonical wire format; matches what an attacker controlling
        # the LLM output would have emitted.
        payload = json.dumps(ctx.input, default=str, ensure_ascii=False)
        try:
            scanned = self._dlp.scan(payload)
        except Exception as exc:
            # fail_closed=True at the hookpoint level means a crashing
            # subscriber is treated as refusal (per HookRegistry semantics).
            # Re-raise to surface the failure loud.
            raise
        if scanned != payload:
            # OutboundDlp.scan returns the redacted text. Any difference
            # means the scan detected a sensitive pattern (canary, API key,
            # credit card, etc.). Refuse the extract.
            from alfred.hooks.errors import HookRefusal
            raise HookRefusal(
                hookpoint="security.quarantined.extract",
                kind="post",
                reason="canary_or_secret_in_extracted_payload",
                subscriber_id=self._SCAN_ID,
            )
        return ctx
```

(Exact `HookRefusal` constructor + `HookContext` shape come from `src/alfred/hooks/`; implementer adapts to the actual API at landing time.)

### 2.4 Subscriber registration — constructor-anchored

The canonical lifecycle anchor for the system-tier DLP subscriber is
`QuarantinedExtractor.__init__`. The extractor owns the trust boundary
the subscriber defends; binding the subscriber's lifecycle to the
extractor's lifecycle keeps "extractor exists ⇒ DLP subscriber is
registered" a structural invariant rather than a bootstrap-ordering
discipline an operator could forget to apply.

The registration helper lives in
`src/alfred/security/_extract_dlp_subscriber.py` and is called from the
extractor's constructor:

```python
# in src/alfred/security/_extract_dlp_subscriber.py

def register_extract_dlp_subscriber(
    *,
    registry: HookRegistry | None = None,
    outbound_dlp: OutboundDlp,
) -> None:
    """Subscribe the DLP scanner to `security.quarantined.extract` post.

    Idempotent on same-instance re-registration: a second call carrying
    the SAME `OutboundDlp` instance is a no-op (first-writer-wins).

    A second call carrying a DIFFERENT `OutboundDlp` instance raises
    `HookError` — silently swapping the active scanner would disarm the
    orchestrator's DLP singleton wiring (the first scanner would remain
    the live defence while the caller's intent was "use *my* scanner").

    Raises `HookError` on a gate-denied or otherwise unavailable
    registration (PRD §7.1 + CLAUDE.md hard rule #7: every
    security-boundary failure is loud). A structlog WARN and audit row
    (`HOOKS_DLP_SUBSCRIBER_NOT_REGISTERED`) emit BEFORE the raise so
    observability survives even if the caller swallows the exception.
    """
    target = registry if registry is not None else get_registry()
    target.register_subscriber(
        hookpoint="security.quarantined.extract",
        kind="post",
        tier="system",
        subscriber_id=OutboundDlpExtractSubscriber._SCAN_ID,
        subscriber=OutboundDlpExtractSubscriber(outbound_dlp=outbound_dlp),
    )
```

`QuarantinedExtractor.__init__` calls this helper directly. Subscriber
lifecycle is bound to extractor existence:

* Multiple extractor instances per process land ONE post-stage
  subscriber on the chain (idempotent helper).
* A process that never constructs a `QuarantinedExtractor` (a tooling
  subprocess, a unit-test fixture that doesn't exercise the dispatch
  chain) has no DLP subscriber registered — which is correct because
  no extract dispatches either.
* A gate-denied or unavailable registration raises out of the
  constructor, so a half-wired extractor cannot exist.

### 2.5 Threat model + residual risk (documented in code)

Inline comment in `OutboundDlpExtractSubscriber.__call__` body documents:

* **Pattern-matchable sub-channel (closed)**: canary tokens, API key regexes, credit card regexes — all caught by `OutboundDlp.scan`.
* **Semantic sub-channel (open — known limitation)**: arbitrary paraphrased PII in natural-language fields is not regex-detectable. Out of scope per current threat model. Future AI-DLP work would close this; tracked separately.

The comment exists so a future reviewer reading the subscriber doesn't conclude "this scan is sufficient defence." It's necessary but not sufficient — and the spec's authors accepted that trade-off when they wrote §6.5 specifying `OutboundDlp.scan` (a regex-based scanner).

## 3. Audit-row interactions

Currently `extract` emits 3 audit-row families directly:

* `quarantine.extract` (success) at line 666
* `quarantine.extract` (typed_refusal) at line 713
* `quarantine.protocol_violation` at line 749

These stay. The hookpoint dispatch ADDS the spec-named `security.quarantined.extract` event flow:

* `pre` audit: subscribers observe schema + handle id (no T3 content)
* `post` audit: subscribers observe the validated `model_dump()` — this is the new defence
* `error` audit: subscribers observe a typed error context (already happens via the existing `protocol_violation` row; the hookpoint adds an additional channel for SIEM-style observers)

**Post-stage refusal audit attribution.** `quarantine.extract` records
the outcome class as `audit_result="post_stage_refused"` for any
post-stage subscriber refusal. The refusing subscriber's identity is
carried on the same row's `refusing_hook_id` field. This pattern exists
because `alfred.hooks.invoke._run_post` does NOT emit `HOOKS_REFUSAL`
audit rows — §6.5's refusal-authorisation contract is pre-stage-only
by design (see `alfred.hooks.invoke` docstrings at lines 519-521 and
282-286). The `quarantine.extract` row is therefore the only forensic
surface for post-stage refusing-subscriber identity. The earlier
`dlp_outbound_refused` token is retained in the closed
`TypedRefusalReason` vocab as a tombstone for forensic-history
continuity; no live emit site uses it. The generic `post_stage_refused`
token + `refusing_hook_id` field combine to future-proof attribution:
any post-stage subscriber's `HookRefusal` will land on this row with
its own `hook_id` without code change. Symmetric-validation in
`AuditWriter.append_schema` forces every emit site for the
`QUARANTINE_EXTRACT_FIELDS` family to populate `refusing_hook_id`
(set to `None` on the success / typed-refusal / transport-failed /
protocol-violation arms).

The other closed-set vocabularies for `result` did NOT need widening
beyond the `post_stage_refused` addition — the hookpoint dispatch
reuses existing audit row shapes via the standard `invoking`
infrastructure.

**Audit-trace consistency (CR-158 round 4).** All `quarantine.*`
audit rows (the `quarantine.extract` row emitted at the post-chain
boundary plus the inline `quarantine.transport_failed` /
`quarantine.protocol_violation` rows the body emits before raising)
AND the pre/post/error hook-chain dispatch rows
`alfred.hooks.invoke.invoke` emits share a single `trace_id` —
`chain_correlation_id`, the UUID minted at the chain-entry point in
`QuarantinedExtractor.extract`. The body-local `correlation_id` (a
per-invocation UUID minted inside `_extract_body`, finer-grained than
the chain id) lives as a field inside the `subject` payload on every
emitted row. Forensic queries join on `trace_id` to walk a single
coherent trace covering every dispatch and audit emission for one
extraction call; `subject.correlation_id` further narrows to a specific
body invocation. The audit helpers
(`_emit_extract_audit`, `_emit_transport_failed_audit`,
`_emit_protocol_violation_audit`) accept `chain_correlation_id` as a
required keyword and assign it to `trace_id`; the body-local
`correlation_id` keeps its existing place in the row's `subject`. A
defence-in-depth fallback is built in: if `chain_correlation_id` is
`None` at the emit site (no such path exists today — every helper is
called from inside the chain — but a future pre-chain validation path
could legitimately introduce one), the helper falls back to the
body-local `correlation_id` so the row still carries a deterministic
trace key rather than an empty field.

### 4.3 Integration — end-to-end DLP-in-chain

`tests/unit/security/test_quarantined_extract_dlp_chain.py`:

* `test_clean_extract_succeeds_with_dlp_subscriber_registered` — happy path.
* `test_extract_with_canary_in_response_is_refused_by_dlp_subscriber` — the quarantined LLM returns an `ExtractionResult` whose `summary` field contains a canary; the post-subscriber raises `HookRefusal`; `extract` propagates the refusal; an audit row is emitted; the validated payload never returns to the caller.
* `test_extract_constructor_raises_when_dlp_subscriber_denied` — fail-loud anchor on the constructor-anchored registration model. `QuarantinedExtractor.__init__` calls `register_extract_dlp_subscriber`; if the helper cannot land a post-stage subscriber on the active `security.quarantined.extract` chain (gate denies the system-tier registration, hookpoint unavailable, etc.), the raise propagates out of `__init__` and the extractor never exists. There is no "still succeeds without DLP subscriber" path — operating without an active post-stage DLP scan on this trust boundary is the failure (PRD §7.1, CLAUDE.md hard rule `#7`).
