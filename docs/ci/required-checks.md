# Required status checks on `main`

This file is the **canonical, version-controlled manifest** of CI checks that must pass before a PR can merge to `main`. The actual enforcement lives in GitHub branch protection (Settings → Branches → `main` → "Require status checks to pass"); this file is the human-readable mirror.

**Why have this file**: branch-protection config is invisible to PRs that should be updating it, not searchable from the repo, and easily out of sync with the YAML that emits the checks. Every change to required checks lands in the same PR as the workflow change that produced them, so the diff is reviewable here.

## How to read the table

- **Check name** — the exact string GitHub matches against the branch-protection list. Comes from the job's `name:` field in the workflow YAML.
- **Workflow** — the file path of the YAML that emits this check.
- **Job key** — the job's YAML key (used in `jobs:`). For navigating the workflow file, not for branch protection.
- **Active since** — the date the check was added to the required list (not the date the workflow was merged).
- **Rationale** — one sentence on why this is required (vs informational).

## How to update this file

When you add a new gating workflow, follow [`.rulesync/skills/author-gating-workflow/SKILL.md`](../../.rulesync/skills/author-gating-workflow/SKILL.md). After the workflow merges and you run `gh api POST .../contexts`, also append a row here in a follow-up PR (or amend the original PR before merge if your branch hasn't been merged yet).

When you remove a required check: same flow in reverse — `gh api -X DELETE .../contexts` and update this file.

### Pending required (promote after green) — arch breadth (#265, ADR-0034)

**Axis 3 — arm64 Linux.** Issue [#265](https://github.com/alfred-os/AlfredOS/issues/265) + the maintainer directive "run on all architectures": these three arm64 jobs mirror their amd64 siblings on `ubuntu-24.04-arm` (free on this public repo) so an architecture-specific regression — native-extension wheel, `ctypes` / `struct` layout, byte-order, POSIX-fd ABI assumption — is caught in CI on arm64 instead of on an operator's Graviton / Apple-Silicon host. Per [ADR-0034](../adr/0034-ci-os-hardware-matrix.md), they were **ADDED as separate jobs with separate check contexts** rather than folding the existing jobs into a `matrix` (a matrix would rename the required amd64 contexts and break branch protection on `main` for every open PR until the protection rules were updated atomically). `Python (lint, types, unit) (arm64)` and `Integration (arm64)` were **promoted to "Currently required" on 2026-06-24** per maintainer directive [#321](https://github.com/alfred-os/AlfredOS/issues/321) (see the "Currently required" table above). The **third** leg — `Integration (privileged Linux, real spawn) (arm64)` — was deferred behind [#269](https://github.com/alfred-os/AlfredOS/issues/269) and is added by that fix; it is pending promotion in the table below.

Two DISTINCT `/lib64` bugs, both now fixed — do not conflate them:

1. **The test-helper bug (fixed earlier, commit `d9f487e1`).** The hand-written bwrap invocation in `tests/integration/test_alfred_core_image_bwrap.py` hard-bound `--ro-bind /lib64` (amd64-only — it holds `ld-linux-x86-64.so.2`); arm64 has no `/lib64`, so the fix binds it only when present. That is what makes the `Integration (arm64)` leg green.
2. **The production-policy bug (#269).** The SHIPPED bwrap policies (`config/sandbox/quarantined-llm.linux.bwrap.policy`, `discord-adapter.linux.bwrap.policy`) ALSO hard-bound `/lib64`, and `policy_to_bwrap_flags` emitted it unconditionally — so the real dual-LLM child died at launch on aarch64 (`bwrap: Can't find source path /lib64`), surfacing as a truncated `read_frame_failed`. This is why the privileged arm64 leg was deferred. The fix adds a `ro_binds_try` policy field emitting `--ro-bind-try` (bind iff present, else skip) and moves `/lib64` to it. The earlier note that the test fix "mirrored the production launcher" was **wrong** — production hard-bound it until #269; it mirrors it now.

Coverage gates are deliberately NOT duplicated on arm64 (coverage is arch-invariant; the amd64 `python` / `coverage-gates` jobs remain the single source of truth — ADR-0034 Decision 2). The arm64 legs run with `--cov-fail-under=0` and `integration-arm64` does NOT upload the `coverage-integration` artifact (a second uploader of that name would collide with amd64's).

## OS matrix — per-OS run/skip reality

The `ci.yml` test layer runs across three operating systems. The split below is the empirically-assessed reality (not an aspiration): each OS runs exactly what is **green** on it today, and POSIX-only tests `skipif`/are-omitted cleanly rather than turning a non-Linux leg red.

| Layer | ubuntu-latest | macos-latest | windows-latest |
| --- | --- | --- | --- |
| Ruff lint + format | ✓ (job `python`) | ✓ (job `python-cross-os`) | ✓ (job `python-cross-os`) |
| mypy --strict + pyright | ✓ | ✓ (Linux-target: `--platform linux` / `--pythonplatform Linux`) | ✓ (Linux-target) |
| Unit suite | ✓ (+ 100% coverage gates) | ✓ blocking (Docker tests auto-skip; #246) | ✓ blocking (Docker + POSIX-only tests skip; #246) |
| Integration / Smoke (Testcontainers) | ✓ | ✗ (Linux-only — Docker daemon) | ✗ (Linux-only) |
| Real-bwrap / launcher-spawn legs | ✓ (job `integration-privileged`, under `sudo`) | n/a | n/a |

Required: **both** the macOS and Windows static-analysis legs are now **required**
(`Python cross-OS (macos-latest)` + `Python cross-OS (windows-latest)`, both
promoted 2026-06-24 — see "Currently required" above). The Windows leg was
informational-until-green (`continue-on-error: ${{ matrix.os == 'windows-latest'
}}`) until [#321](https://github.com/alfred-os/AlfredOS/issues/321) Phase 3
dropped the `continue-on-error` after 6 consecutive green cycles (#312–#319);
the leg now blocks like macOS. (The static-analysis layer is required on both non-Linux legs; the unit suite
is now BLOCKING on macOS and Windows — #246 Phase A/B. See below.)

**Why each OS runs what it runs:**

- **ubuntu-latest** — the full battle-tested layer. The dedicated `python` job keeps its name and its per-subsystem 100% coverage gates (those need the combined unit+integration corpus the Linux `coverage-gates` job assembles, so they are deliberately NOT duplicated on the other OSes). The new `integration-privileged` job adds the root+bwrap depth (Axis 2).
- **macos-latest** — `lint + format + mypy + pyright`, plus the unit suite (blocking; #246 Phase A). Type-checking is pinned to the Linux deployment target (`mypy --platform linux` / `pyright --pythonplatform Linux`): the runtime is Linux-only (bwrap / runuser / POSIX fds) and uses POSIX-only `os` attributes, so the right thing to type-check against is the Linux stubs the code actually runs on, on every runner OS. macOS = bash 3.2 + BSD coreutils, so this leg is the real portability signal for macOS-specific path / type drift. The unit suite **now runs here** (#246 Phase A): the 4 Testcontainers unit files (`tests/unit/plugins/web_fetch/`) are module-marked `pytest.mark.docker` and the root-conftest collection hook skips them cleanly when no Docker daemon is reachable (the macOS runner has none), so the rest of `tests/unit` runs and gives the Darwin-vs-Linux divergence signal (e.g. the class of bug the #340 PR2a `recvmsg`/`MSG_CTRUNC` split hit). An anti-rot guard (`tests/unit/meta/test_docker_tests_are_marked.py`) fails if a new Testcontainers file is unmarked, so the separation can't rot. No coverage gate here — the per-subsystem 100% gates stay on the Linux `python` job.
- **windows-latest** — `lint + format + mypy + pyright` (same layer as macOS) plus the unit suite (**blocking**; #246 Phase B), likewise type-checked against the Linux target (`--platform linux` / `--pythonplatform Linux`) so the runtime's POSIX-only `os` surface does not raise false `attr-defined` errors against the Windows `os` stubs. The unit suite **runs BLOCKING here** (#246 Phase B, 2026-07-12) over the portable subset (~5.4k tests): Docker-marked tests auto-skip on win32 (Docker Desktop reports available but the Linux-container Testcontainers can't run — the `/var/run/docker.sock` bind is invalid), the wholly-POSIX unit files are collect-ignored on win32 (`tests/_posix_only_tests.py` — bash/`runuser` launcher, `AF_UNIX` daemon sockets, POSIX pidfile mode), and the POSIX-only tests inside otherwise-portable files carry `skipif(sys.platform == "win32")` (`os.getuid` family, `os.writev`/`mkfifo`/`killpg`, `socket.AF_UNIX`, POSIX file mode, `os.O_NOFOLLOW` via the daemon-boot path). Windows dev of the Linux-only surface is via WSL2; the win32 guards were built discover-then-guard against the real CI failure surface (the dev box is Darwin). The Windows static-analysis layer (lint/format/type) is **required** — the job-level `continue-on-error` gating it was removed 2026-06-24 (#321 Phase 3) after 6 green cycles; the unit step's own step-level `continue-on-error` was dropped 2026-07-12 (#246 Phase B).

**Cost discipline.** A 3-OS matrix would triple minutes if applied naively. Instead: the heavy Testcontainers Integration/Smoke jobs stay Linux-only; only the cheap static-analysis layer fans out to macOS + Windows. `fail-fast: false` on the `python-cross-os` matrix so a macOS failure does not mask a Windows failure (or vice-versa).

**Deferred to a follow-up issue (#246, links #245):**

1. **Docker-free unit subset on macOS — DONE (#246 Phase A).** Shipped via the `docker` marker + root-conftest auto-skip hook (not the originally-sketched `requires_docker` / `-m "not requires_docker"` deselect); the macOS leg now runs `pytest tests/unit -q` blocking-green with the Testcontainers files skipping.
2. **Unit suite on Windows → BLOCKING — DONE (#246 Phase B, 2026-07-12).** Shipped via a three-part win32 guard: docker auto-skip on win32 + collect-ignore of the wholly-POSIX unit files (`tests/_posix_only_tests.py`) + `skipif(sys.platform == "win32")` on the POSIX-only tests in mixed files; then the step-level `continue-on-error` was dropped. The Windows unit leg now blocks over the portable ~5.4k-test subset (`5459 passed, 401 skipped` on promotion). Branch-protection contexts unchanged — `Python cross-OS (windows-latest)` was already required, so no `gh api .../contexts` call was needed.
3. **macOS-native `sandbox-exec` execution** — once PR-S4-7 ships the real `sandbox-exec` invocation for `kind=full`, the macOS leg should run those sandbox legs natively. Linux already runs the native bwrap path via `integration-privileged`.

## On bypass

Pre-push hooks (`lefthook`) can be skipped with `LEFTHOOK=0 git push`. This bypass is local-only — **the required status checks listed above cannot be bypassed**; they run on every PR and the merge button is blocked until they pass. Treat `LEFTHOOK=0` as functionally equivalent to `--no-verify`: emergency-only, never a habit.
