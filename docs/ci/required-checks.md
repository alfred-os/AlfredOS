# Required status checks on `main`

This file is the **canonical, version-controlled manifest** of CI checks that must pass before a PR can merge to `main`. The actual enforcement lives in GitHub branch protection (Settings → Branches → `main` → "Require status checks to pass"); this file is the human-readable mirror.

**Why have this file**: branch-protection config is invisible to PRs that should be updating it, not searchable from the repo, and easily out of sync with the YAML that emits the checks. Every change to required checks lands in the same PR as the workflow change that produced them, so the diff is reviewable here.

## How to read the table

- **Check name** — the exact string GitHub matches against the branch-protection list. Comes from the job's `name:` field in the workflow YAML.
- **Workflow** — the file path of the YAML that emits this check.
- **Job key** — the job's YAML key (used in `jobs:`). For navigating the workflow file, not for branch protection.
- **Active since** — the date the check was added to the required list (not the date the workflow was merged).
- **Rationale** — one sentence on why this is required (vs informational).

## How to update this file

When you add a new gating workflow, follow [`.rulesync/skills/author-gating-workflow/SKILL.md`](../../.rulesync/skills/author-gating-workflow/SKILL.md). After the workflow merges and you run `gh api POST .../contexts`, also append a row here in a follow-up PR (or amend the original PR before merge if your branch hasn't been merged yet).

When you remove a required check: same flow in reverse — `gh api -X DELETE .../contexts` and update this file.

>    and short-circuits to success on off-path PRs.
> 2. Drop the `paths:` filter from `perf.yml` (lets the workflow run on all PRs;
>    higher CI cost, but simpler).
> 3. Use `actions/changed-files` inside a single job to gate the actual bench run
>    while always reporting the check name.

### Pending required (promote after green) — arch breadth (#265, ADR-0034)

**Axis 3 — arm64 Linux.** Issue [#265](https://github.com/alfred-os/AlfredOS/issues/265) + the maintainer directive "run on all architectures": these three arm64 jobs (added together) mirror their amd64 siblings on `ubuntu-24.04-arm` (free on this public repo) so an architecture-specific regression — native-extension wheel, `ctypes` / `struct` layout, byte-order, POSIX-fd ABI assumption — is caught in CI on arm64 instead of on an operator's Graviton / Apple-Silicon host. Per [ADR-0034](../adr/0034-ci-os-hardware-matrix.md), they were **ADDED as separate jobs with separate check contexts** rather than folding the existing jobs into a `matrix` (a matrix would rename the required amd64 contexts and break branch protection on `main` for every open PR until the protection rules were updated atomically). Both `Python (lint, types, unit) (arm64)` and `Integration (arm64)` were **promoted to "Currently required" on 2026-06-24** per maintainer directive [#321](https://github.com/alfred-os/AlfredOS/issues/321) (see the "Currently required" table above); the section below is retained for the ADR-0034 rationale and the still-deferred third arm64 leg.

The bwrap `/lib64` arch-portability fix (commit `d9f487e1` on this branch) is what makes the `Integration (arm64)` leg green: the hand-written bwrap invocation in `tests/integration/test_alfred_core_image_bwrap.py` hard-bound `--ro-bind /lib64` (amd64-only — it holds `ld-linux-x86-64.so.2`); arm64 has no `/lib64`, so the fix binds it only when present, mirroring the production launcher.

Coverage gates are deliberately NOT duplicated on arm64 (coverage is arch-invariant; the amd64 `python` / `coverage-gates` jobs remain the single source of truth — ADR-0034 Decision 2). The arm64 legs run with `--cov-fail-under=0` and `integration-arm64` does NOT upload the `coverage-integration` artifact (a second uploader of that name would collide with amd64's).

Both arm64 legs that were pending here (`Python (lint, types, unit) (arm64)`, `Integration (arm64)`) are now in the "Currently required" table (promoted 2026-06-24, [#321](https://github.com/alfred-os/AlfredOS/issues/321)).

A third arm64 leg, `Integration (privileged Linux, real spawn) (arm64)` (the dual-LLM real-`bwrap` spawn), is **deferred**: its first arm64 run caught a real arm64 real-spawn bug ([#269](https://github.com/alfred-os/AlfredOS/issues/269), overlapping [#252](https://github.com/alfred-os/AlfredOS/issues/252)). It is added — and promoted — with the #269 fix, not here, to avoid landing a red leg on `main`.

| Unit suite | ✓ (+ 100% coverage gates) | ✗ deferred (Docker-needing tests; see below) | ✗ deferred (see below) |
| Integration / Smoke (Testcontainers) | ✓ | ✗ (Linux-only — Docker daemon) | ✗ (Linux-only) |
| Real-bwrap / launcher-spawn legs | ✓ (job `integration-privileged`, under `sudo`) | n/a | n/a |

Required-vs-informational: the macOS leg is **required** (`Python cross-OS
(macos-latest)`, promoted 2026-06-24 — see "Currently required" above); the
Windows leg is **informational-until-green** (`continue-on-error: ${{
matrix.os == 'windows-latest' }}` — visible but non-blocking, still in "Pending
required"). [#321](https://github.com/alfred-os/AlfredOS/issues/321) Phase 3 will
drop the `continue-on-error` and promote Windows once the leg is confirmed stable.

**Why each OS runs what it runs:**

- **ubuntu-latest** — the full battle-tested layer. The dedicated `python` job keeps its name and its per-subsystem 100% coverage gates (those need the combined unit+integration corpus the Linux `coverage-gates` job assembles, so they are deliberately NOT duplicated on the other OSes). The new `integration-privileged` job adds the root+bwrap depth (Axis 2).
- **macos-latest** — `lint + format + mypy + pyright` only. Type-checking is pinned to the Linux deployment target (`mypy --platform linux` / `pyright --pythonplatform Linux`): the runtime is Linux-only (bwrap / runuser / POSIX fds) and uses POSIX-only `os` attributes, so the right thing to type-check against is the Linux stubs the code actually runs on, on every runner OS. macOS = bash 3.2 + BSD coreutils, so this leg is the real portability signal for macOS-specific path / type drift. The unit suite is **not** run here: ~77 tests under `tests/unit/` boot a real Redis/Postgres via Testcontainers (no `@pytest.mark.integration` — they `import testcontainers` directly) and the GitHub macOS runner has no Docker daemon, so they ERROR at fixture setup (`docker.errors.DockerException`; #245 first real run = 3731 passed / 77 errored). They are not cleanly separable by a single marker or path, so a deselect allowlist would rot the moment a new Docker-backed unit test lands — the Docker-free macOS unit subset is deferred (see below).
- **windows-latest** — `lint + format + mypy + pyright` only, same layer as macOS and likewise type-checked against the Linux target (`--platform linux` / `--pythonplatform Linux`) so the runtime's POSIX-only `os` surface does not raise false `attr-defined` errors against the Windows `os` stubs. The unit suite is **not** Windows-clean: many tests invoke the POSIX bash launcher (`/bin/sh`), `os.uname()`, `os.geteuid()`, `signal.SIGKILL`, and `subprocess(..., pass_fds=...)` with no `sys.platform == "win32"` skip-guard, so they would ERROR rather than skip. Adding those guards is a cross-cutting code change across dozens of files — out of scope for this additive CI PR, tracked as a follow-up to #245. The Windows leg is informational (`continue-on-error`): visible but non-blocking until it burns in.

**Cost discipline.** A 3-OS matrix would triple minutes if applied naively. Instead: the heavy Testcontainers Integration/Smoke jobs stay Linux-only; only the cheap static-analysis layer fans out to macOS + Windows. `fail-fast: false` on the `python-cross-os` matrix so a macOS failure does not mask a Windows failure (or vice-versa).

**Deferred to a follow-up issue (#246, links #245):**

1. **Docker-free unit subset on macOS** — the ~77 Testcontainers-backed tests under `tests/unit/` need a `requires_docker` marker so the macOS leg can run `pytest tests/unit -m "not requires_docker"` and add real unit coverage on Darwin (bash 3.2 / BSD coreutils / `os.uname()`/`geteuid()` paths) without erroring on the absent Docker daemon.
2. **Unit suite on Windows** — requires `sys.platform == "win32"` skip-guards (or pure-Python reimplementations) on the bash-launcher / POSIX-syscall unit tests so they skip cleanly instead of erroring; then the Windows leg can be promoted from informational to required.
3. **macOS-native `sandbox-exec` execution** — once PR-S4-7 ships the real `sandbox-exec` invocation for `kind=full`, the macOS leg should run those sandbox legs natively. Linux already runs the native bwrap path via `integration-privileged`.

## On bypass

Pre-push hooks (`lefthook`) can be skipped with `LEFTHOOK=0 git push`. This bypass is local-only — **the required status checks listed above cannot be bypassed**; they run on every PR and the merge button is blocked until they pass. Treat `LEFTHOOK=0` as functionally equivalent to `--no-verify`: emergency-only, never a habit.
